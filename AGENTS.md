# AGENTS.md

## 목적

이 폴더는 POSCO DX AI Youth Challenge 제출 아이디어를 축적하고, 신청서와 실제 MVP 개발까지 이어가기 위한 작업공간이다. 문서는 "좋은 말"보다 **개발 가능한 범위, 근거, 데모 흐름, 안전장치**가 남는 것을 우선한다.

현재 폴더에는 두 아이디어 방향이 섞여 있다. 새 작업을 시작할 때는 먼저 어느 방향을 기준으로 하는지 확인한다.

- **현재 개발 기준안**: `01_Idea/idea_.md`의 "외국인 근로자를 위한 AI 근로계약서 도우미"
- **이전/대안 방향**: 경계선 지능 청소년·청년을 위한 쉬운 말 계약 이해 보조 서비스

사용자가 `idea_.md`, MVP 개발, Expo, FastAPI, 외국인 근로자, 다국어 설명을 말하면 현재 개발 기준안을 따른다. 사용자가 경계선 지능, 쉬운 정보, 아르바이트 청소년·청년을 말하면 이전/대안 방향으로 다룬다. 두 방향을 한 문서 안에서 섞지 않는다.

## 현재 기준 자료

- `01_Idea/idea_.md`: 현재 개발 기준안. 외국인 근로자 대상, React Native + Expo Android 앱, FastAPI 백엔드, OCR, OpenAI API 흐름을 담고 있다.
- `01_Idea/2026-05-31_idea_review.md`: `idea_.md`의 점검 결과. 개발 전에 고쳐야 할 충돌, 위험, 빈칸을 확인한다.
- `03_Working/PRD.md`: 개발 기준 요구사항 문서. 개발할 때 가장 먼저 확인한다.
- `03_Working/roadmap.md`: 단계별 개발 로드맵. 작업 순서와 완료 기준을 확인한다.
- `03_Working/progress.json`: 현재 진행상태 추적 파일. 완료/진행/보류 상태를 기계적으로 갱신한다.
- `03_Working/2026-05-30_쉬운계약_MVP_개발플랜.md`: 이전 방향의 상세 개발 플랜. 현재 개발 기준안과 충돌하는 내용은 그대로 가져오지 말고 참고만 한다.
- `02_References/POSCO DX AI Youth Challenge_AWARD_REFERENCE.md`: 2021-2025년 수상사례 정리 문서. 아이디어를 따라 하기보다 대회가 선호하는 문제 정의, AI 활용, 구현 구체성, 사회적 효과 서술 방식을 참고한다.
- `02_References/Report Reference.pdf`: 신청서 작성 예시. 문장이나 구조를 복사하지 않고, 최종 신청서의 파트 구성과 서술 밀도를 참고한다.

## 작업 원칙

1. 아이디어를 "AI 법률 판정기"로 만들지 않는다.
   - 법률 판단, 위법 여부 판정, 계약 무효 판단을 AI가 대신한다고 쓰지 않는다.
   - 기본 표현은 "근로계약 이해 보조", "주의 신호 안내", "스스로 확인할 수 있도록 돕는 도구"로 잡는다.

2. 대상 사용자를 좁혀서 쓴다.
   - 현재 개발 기준안: 한국어 근로계약서를 혼자 확인해야 하는 외국인 근로자
   - 이전/대안 방향: 혼자 아르바이트 근로계약서를 확인해야 하는 경계선 지능 청소년·청년
   - 한 번에 모든 계약서, 모든 취약계층, 금융·부동산·보험까지 확장하지 않는다.

3. 보호주의적 표현을 피한다.
   - "대신 판단해준다"보다 "이해를 돕는다", "확인 질문을 제공한다", "필요할 때 상담기관과 연결한다"로 쓴다.
   - 사용자의 자기결정권을 해치지 않는 방향으로 설명한다.

4. AI가 필요한 이유를 구체화한다.
   - 단순 키워드 검색이나 번역 앱처럼 보이면 약하다.
   - 핵심 AI 기능은 OCR 결과 정리, 핵심 근로조건 추출, 쉬운 말 변환, 의미 기반 주의 조항 분류, 다국어 설명 생성으로 정리한다.

5. 근거가 없으면 지어내지 않는다.
   - 통계, 피해 사례, 기관명, 법률 기준, 상담 전화번호는 출처가 확인될 때만 쓴다.
   - 아직 확인하지 않은 내용은 `TODO: 근거 확인 필요`로 남긴다.

6. 새 자료는 원본과 해석을 구분한다.
   - 기사, 논문, 기관 자료는 요약만 하지 말고 "우리 아이디어에 주는 의미"를 함께 남긴다.
   - 수상사례는 베끼지 않고 평가 관점과 구조만 참고한다.

7. 개발 문서는 추적 가능하게 쓴다.
   - 범위 변경은 `PRD.md`와 `roadmap.md`에 반영한다.
   - 상태 변경은 `progress.json`의 `status`, `updatedAt`, `notes`, `blockers`를 갱신한다.
   - 큰 결정은 `03_Working/YYYY-MM-DD_결정명.md`로 남긴다.

## 추천 폴더 구조

```text
.
├── AGENTS.md
├── 01_Idea/
│   └── 현재 아이디어, 문제 정의, 페르소나, 사용 시나리오, 점검 문서
├── 02_References/
│   └── 수상사례, 신청서 예시, 통계, 기사, 논문, 기관 자료
├── 03_Working/
│   └── PRD, roadmap, progress, 제출용 초안, 기능 정리, 화면 흐름
└── 99_Archive/
    └── 더 이상 쓰지 않는 구버전 자료
```

- 아이디어 자체를 바꾸는 문서: `01_Idea/`
- 외부 참고자료와 우수사례: `02_References/`
- 실제로 쓰는 초안과 작업본: `03_Working/`
- 버리긴 아깝지만 현재 방향과 다른 파일: `99_Archive/`

## 새 문서 작성 규칙

- 파일명은 가능하면 `YYYY-MM-DD_주제.md` 형식으로 쓴다.
- 계속 갱신되는 개발 기준 문서는 예외적으로 `PRD.md`, `roadmap.md`, `progress.json`처럼 고정 이름을 쓴다.
- 초안은 완성된 말투보다 판단 근거가 남는 것이 중요하다.
- 제출용 문서가 아니면 과하게 꾸미지 말고 짧고 명확하게 쓴다.
- 학생들이 바로 고칠 수 있게 문장은 길게 늘리지 않는다.

## 신청서 초안으로 옮길 때의 기준

`02_References/Report Reference.pdf`의 신청서 예시는 다음 관점에서만 참고한다.

- Part 1 아이디어: 제안 배경, 차별성, 핵심 내용, AI 필요성, 긍정적 효과
- Part 2 프로그램: 개발 분야, 사용 기술, 프로토타입 설계, 주요 기능
- Part 3 계획안: 팀원 역할, 월별 활동계획, 구현 일정

최종 신청서에서는 우수사례의 문장이나 구조를 그대로 복사하지 않는다. 현재 개발 기준안에 맞게 "한국어 근로계약서를 이해하기 어렵거나 놓치기 쉬운 사용자가, 계약조건과 주의 조항을 쉽게 확인하도록 돕는 AI 서비스"로 다시 쓴다.

## 현재 개발 기준안의 안전한 한 문장

> 외국인 근로자가 한국어 근로계약서를 촬영하면, OCR과 AI가 핵심 근로조건과 주의해야 할 조항을 쉬운 한국어와 모국어로 설명해 주는 근로계약 이해 보조 서비스.

## 앞으로 우선 쌓아야 할 자료

- 외국인 근로자의 한국어 근로계약서 이해 어려움 관련 자료
- 근로계약서에서 자주 문제가 되는 항목: 임금, 근로시간, 휴일, 수습기간, 숙식비 공제, 위약금, 퇴직금 등
- 표준근로계약서 양식과 필수 기재 항목
- 개인정보 보호와 법률 조언 오해를 피하기 위한 안내 문구
- 번역 앱, 일반 계약서 요약 서비스, 노무 상담 서비스와의 차별점
