# Phase 10 사진 출처 하이라이트 설계 메모

> 상태: 방향 확정 메모. Phase 9 다국어 안정화 후 구현.
> 관련: PRD F-17, roadmap Phase 10, progress M8
> 작성 이유: 결과 카드가 "AI가 이렇게 말함"에 그치면 사용자가 믿기 어렵다. 주의 조항이 실제 계약서 사진의 어디에서 온 것인지 확인할 수 있어야 한다.

## 핵심 판단

이 기능은 단순 UI 장식이 아니라 신뢰 UX다.

현재 결과 카드는 `originalText` 인용문을 보여주지만, 사용자는 여전히 "이 문장이 사진 어디에 있었지?"를 직접 찾아야 한다. 특히 OCR이 틀리거나 AI가 잘못 짚는 경우를 사용자가 검증할 방법이 약하다.

따라서 Phase 10의 목표는 다음으로 잡는다.

- 주의 카드마다 **사진에서 보기** 진입을 제공한다.
- 누르면 계약서 원본 이미지가 열리고, 해당 문장/값이 나온 위치를 하이라이트한다.
- 매칭 신뢰도가 낮으면 "원문 위치를 정확히 찾기 어려워요"라고 표시한다.
- AI가 좌표를 만들지 않는다. 좌표는 OCR 결과와 서버 매칭 로직이 계산한다.

## 권장 UX

### 1차 핵심 플로우

1차 구현은 `사진에서 확인` 하나에 집중한다. overview나 전체 문서 맵보다 먼저, 사용자가 결과 카드 하나를 눌렀을 때 "이 분석이 실제 사진의 이 부분에서 왔다"를 바로 확인하게 만드는 것이 목표다.

```text
결과 카드
→ 사진에서 확인
→ 해당 페이지 원본 이미지
→ OCR 근거 영역 하이라이트
→ 하단 근거 패널: OCR 원문 + AI 설명
```

이 흐름이 먼저 안정되면, 사용자는 결과를 맹신하지 않고 원문과 비교할 수 있다. 반대로 overview를 먼저 만들면 화면은 화려하지만 "내가 지금 확인하려는 항목이 어디인지"가 흐려질 수 있다.

### 결과 카드

각 `CautionCard` 구성:

1. 레벨 배지
2. 제목
3. 계약서 원문 인용
4. 설명
5. `사진에서 확인` 버튼

`사진에서 확인`은 매칭 성공 시 primary/blue text action으로 둔다. 매칭 실패 시에는 억지로 버튼을 보여주지 않고, 카드 하단에 `원문 위치를 정확히 찾기 어려워요`를 작게 표시한다.

카드 문구 예:

```text
임금 비정상 기재

계약서 원문
"월 통상임금(100)원"

임금이 100원으로 적혀 있어 실제 지급액인지 확인이 필요합니다.

사진에서 확인
```

### 이미지 뷰어

`사진에서 확인`을 누르면:

- 원본 이미지 뷰어를 연다.
- 해당 페이지로 이동한다.
- OCR 매칭 박스를 반투명 파란색으로 표시한다.
- 하단에는 카드 제목, OCR 원문, AI 설명을 짧게 보여준다.
- 여러 박스가 있으면 같은 문장 단위로 묶어 표시한다.
- 핀치 줌은 있으면 좋지만 1차 구현 필수는 아니다. 먼저 "어디쯤인지 보이는 것"을 우선한다.

권장 레이아웃:

```text
┌────────────────────────────┐
│ ←  사진에서 확인        1/2 │
├────────────────────────────┤
│                            │
│       계약서 원본 이미지       │
│        ▣ 하이라이트 박스       │
│                            │
├────────────────────────────┤
│ 임금 비정상 기재              │
│ OCR 원문                     │
│ "월 통상임금(100)원"          │
│                              │
│ 이 부분을 근거로 분석했어요.    │
└────────────────────────────┘
```

하이라이트 색은 Toss 톤을 유지해 파란색 계열을 기본으로 한다. 위험/확정 판단처럼 보이지 않게 빨간색 박스는 피한다. 빨간색은 `확인 필요` 배지처럼 결과 카드의 상태 표현에만 제한적으로 사용한다.

### 하단 근거 패널

이미지 뷰어 하단에는 법률 판단보다 근거 연결을 강조한다.

표시 순서:

1. 카드 제목
2. `OCR 원문`
3. 원문 인용
4. `이 부분을 근거로 분석했어요.`
5. 짧은 설명

주의 문구:

- `이 조항은 위법입니다` 금지
- `이 위치가 정확합니다` 금지
- `이 부분을 근거로 분석했어요` 권장
- `OCR 위치가 다를 수 있어 원문을 함께 확인해주세요` 낮은 신뢰도에서 표시

### 실패/낮은 신뢰도

OCR 텍스트가 깨졌거나 문장 매칭이 약하면:

- 사진 하이라이트를 억지로 찍지 않는다.
- 카드에는 원문 인용만 보여준다.
- `사진에서 확인` 대신 `원문 위치를 정확히 찾기 어려워요` 안내를 둔다.

낮은 신뢰도에서 이미지 뷰어를 열 수 있게 할 경우에는 하이라이트를 점선/연한 박스로 표시하고, 하단에 `OCR 위치가 정확하지 않을 수 있어요`를 반드시 붙인다. 1차 구현에서는 낮은 신뢰도 항목은 뷰어 진입을 막는 편이 더 안전하다.

## 2차 확장: 문서 근거 overview

`사진에서 확인`이 안정된 뒤에는 결과 상단에 `계약서에서 찾은 주요 영역` overview를 추가할 수 있다.

목표:

- 사용자가 계약서 전체에서 어떤 영역이 읽혔는지 한눈에 본다.
- 임금, 근로시간, 휴일, 계약기간, 숙식비/공제, 확인 필요한 조항을 페이지 썸네일 위에 표시한다.
- overview 영역을 누르면 해당 결과 카드 또는 이미지 뷰어로 이동한다.

1차에서 제외하는 이유:

- 여러 항목을 한 사진에 동시에 표시하면 화면이 복잡해질 수 있다.
- 지금 가장 큰 문제는 "개별 분석 결과의 근거를 믿기 어렵다"이므로, 카드별 근거 확인이 더 직접적이다.
- overview는 데이터 매칭 품질이 쌓인 뒤 넣어야 잘못된 박스가 한꺼번에 보이는 위험을 줄일 수 있다.

## 데이터 구조

AI가 좌표를 직접 생성하면 안 된다. AI는 `originalText` 또는 `sourceQuote`만 준다. 서버가 OCR 좌표와 문장 매칭으로 위치를 계산한다.

권장 추가 필드:

```ts
type SourceBox = {
  pageIndex: number;
  x: number;      // 이미지 원본 너비 기준 0~1
  y: number;      // 이미지 원본 높이 기준 0~1
  width: number;  // 0~1
  height: number; // 0~1
};

type SourceMatch = {
  pageIndex: number;
  quote: string;
  boxes: SourceBox[];
  confidence: 'high' | 'medium' | 'low';
  matchType: 'exact' | 'normalized' | 'fuzzy';
};

type CautionItem = {
  level: CautionLevel;
  title: LocalizedText;
  originalText: string;
  explanation: LocalizedText;
  source?: SourceMatch;
};
```

좌표는 저장과 화면 크기 변화에 강하게 하기 위해 픽셀값보다 정규화 좌표(0~1)를 쓴다.

`SourceBox`는 OCR 원본 이미지 기준 좌표다. 앱에서는 이미지가 `contain`으로 렌더링될 때 생기는 좌우/상하 여백을 계산한 뒤 박스를 그려야 한다. 단순히 화면 width/height에 바로 곱하면 이미지 비율이 다른 경우 박스가 어긋난다.

## 서버 구현 방향

1. Google Vision OCR 응답에서 페이지별 word/block bounding box를 보존한다.
2. OCR 전체 텍스트는 기존처럼 AI 분석에 보낸다.
3. AI가 반환한 `cautionItems[].originalText`를 OCR 단어/문장 목록과 매칭한다.
4. 정규화된 박스 배열과 confidence를 `cautionItems[].source`에 붙인다.
5. 저장 모델은 새 필드를 그대로 저장한다.

매칭 방식:

- 1차: normalized exact substring
- 2차: 공백/기호 제거 후 substring
- 3차: token overlap/fuzzy matching
- 실패: `source` 없음 또는 `confidence: low`

### 매칭 정책

매칭 대상은 AI 설명문이 아니라 `originalText/sourceQuote`다. 설명문은 선택 언어로 번역될 수 있으므로 OCR 원문과 직접 매칭하면 안 된다.

권장 기준:

- `high`: 원문 quote가 OCR line 또는 연속 word에 거의 그대로 포함됨
- `medium`: 공백/괄호/기호 제거 후 포함됨
- `low`: 토큰 일부만 겹치거나 페이지/영역 추정만 가능함
- `none`: 매칭 실패. 앱에 `source`를 보내지 않음

서버는 `low`를 보내더라도 앱이 1차 구현에서 버튼을 숨길 수 있게 해야 한다. 사용자가 틀린 하이라이트를 더 신뢰하는 일이 생기면 기능 목적이 반대로 무너진다.

### OCR 품질 체크와 연결

사진 자체를 자동 스캔하지 않더라도, OCR 품질이 낮으면 이 기능이 실패한다. 따라서 Phase 10 구현 시 서버에서 다음 신호를 같이 계산한다.

- 전체 OCR 글자 수가 너무 적음
- 근로계약서 핵심 키워드가 거의 없음
- 페이지별 OCR box 수가 너무 적음
- 같은 문장이 너무 넓은 영역에 흩어져 있음

이 경우 앱은 분석 결과 전에 재촬영을 강제하지 않더라도, 결과 화면에 `문서가 선명하지 않아 일부 위치를 찾기 어려울 수 있어요`를 표시한다.

## 범위

### 1차 구현

- 서버 OCR 좌표 반환
- 조항 원문과 OCR 위치 매칭
- 결과 카드의 `사진에서 확인`
- 이미지 뷰어 오버레이 하이라이트
- 낮은 신뢰도 안내
- 새 기록에 source 저장

### 제외

- 기존 저장 기록에 좌표 소급 적용
- 완벽한 문장 매칭 보장
- 법률 판단 자동화
- OCR 오류 자동 수정
- 복잡한 다중 페이지 비교 UI
- iPhone 메모 앱 수준의 자동 문서 스캔/자동 크롭
- 모든 요약 항목의 위치 표시. 1차는 `cautionItems`부터 적용

## 주의점

- 이 기능은 "AI를 믿으세요"가 아니라 "사용자가 직접 확인하세요"를 강화한다.
- 하이라이트가 틀릴 수 있으므로 confidence 표시와 실패 안내가 필요하다.
- 원문 이미지는 민감 정보이므로 서버에 저장하지 않는 원칙은 유지한다.
- 좌표와 source match는 앱 로컬 기록에만 저장한다.

## QA 기준

다음 테스트를 통과해야 Phase 10을 완료로 본다.

1. 실제 계약서 2장 업로드 후, 각 주의 카드의 `사진에서 확인`이 올바른 페이지로 이동한다.
2. `월 통상임금(100)원`처럼 짧은 값도 해당 줄 근처에 하이라이트된다.
3. 긴 조항은 여러 단어 박스를 하나의 근거 묶음처럼 표시한다.
4. 매칭 실패 항목은 하이라이트를 억지로 표시하지 않는다.
5. 이미지가 화면에 `contain`으로 들어가도 박스 위치가 어긋나지 않는다.
6. 저장 후 다시 열어도 source 하이라이트가 유지된다.
7. 기존 저장 기록은 source가 없어도 결과 화면이 깨지지 않는다.
