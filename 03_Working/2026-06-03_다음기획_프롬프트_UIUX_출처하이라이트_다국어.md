## 1. 왜 지금 이 순서인가

현재 앱은 OCR→AI→결과 표시→저장·재열람→언어 설정까지 연결됐다. 이제 남은 문제는 기능 추가보다 **분석 신뢰도와 결과 검증 가능성**이다.

사용자 테스트에서 확인된 다음 이슈를 다음 기획의 기준으로 잡는다.

1. AI 분석 프롬프트를 더 안정적으로 조율해야 한다.
2. 결과 화면과 원본 사진을 함께 보며 "이 분석이 사진의 어느 부분에서 나온 것인지" 확인할 수 있어야 한다.
3. 결과 화면 상단 언어 전환에서 한국어/영어는 잘 보이지만, 그 외 언어는 UI 라벨만 바뀌고 분석 결과 동적 내용은 영어로 fallback 되는 문제가 있다.
4. UI/UX는 결과 신뢰 UX 중심으로 다시 정비해야 한다.

중요한 판단:

- `사진에서 확인` 기능은 UI만으로 만들 수 없다. AI가 `originalText`를 정확히 짧게 인용하고, 서버가 OCR 좌표와 매칭해야 한다.
- 다국어 문제는 번역 문구 부족이 아니라, 분석 결과 JSON에 해당 언어 키가 없을 때 앱이 영어로 fallback 하는 구조 문제다.
- 프롬프트 품질, 다국어 구조, 사진 출처 하이라이트는 서로 연결되어 있으므로 순서를 나눠야 한다.

## 2. 현재 버그/관찰 기록

### 동적 다국어 fallback 문제

현재 서버 분석은 선택 언어 기준으로 `ko`, `en`, `요청언어`만 생성한다.

예:

- 분석 요청 시 언어가 `ko`면 결과 JSON은 보통 `ko`, `en`만 가진다.
- 결과 화면에서 이후 `ne`, `vi`, `km` 같은 언어를 누르면 해당 key가 없다.
- 앱의 `getLocalized()`는 `value[language] ?? value.en ?? value.ko` 순서라서 동적 분석 내용은 영어로 보인다.
- 그래서 카테고리/라벨/UI 문구는 번역되지만, AI가 만든 요약값·주의 조항 설명은 영어로 남는다.

이 문제는 다음 중 하나로 해결해야 한다.

| 선택지 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 분석 시 16개 언어 모두 생성 | 결과 화면 전환 즉시 반영 | 토큰 비용·응답 시간 증가, JSON 실패 위험 증가 | 데모용으로는 과함 |
| 결과 탭을 `ko/en/분석요청언어`만 노출 | 구현 쉬움 | 사용자가 결과에서 원하는 언어로 바꾸는 요구를 충족하지 못함 | 임시 방편 |
| 언어 전환 시 없는 언어만 서버에 번역 요청 후 캐시 | 비용·시간 균형, 저장 기록도 보강 가능 | 새 API와 로딩 상태 필요 | 권장 |

권장 방향은 **on-demand localization cache**다. 결과 화면에서 사용자가 언어를 바꿨을 때 해당 언어 key가 없으면 서버에 보강 요청을 보내고, 받은 번역을 현재 session/저장 record에 병합한다.

### 사진 출처 하이라이트의 전제

사진 하이라이트는 AI가 좌표를 직접 만드는 방식으로 구현하면 안 된다. 좌표는 Google Vision OCR 결과의 bounding box에서 나오고, 서버가 `originalText` 또는 `sourceQuote`와 OCR line/word를 매칭해 계산한다.

따라서 프롬프트는 다음을 반드시 지켜야 한다.

- `originalText`는 OCR에 실제로 있는 한국어 원문만 짧게 인용한다.
- 설명문 번역과 원문 인용을 섞지 않는다.
- 원문이 길면 하이라이트 가능한 핵심 phrase를 `sourceQuote`로 분리하는 방안을 검토한다.
- 매칭이 낮은 경우 앱은 버튼을 숨기거나 낮은 신뢰도 안내를 보여준다.

## 3. 다음 구현 순서

### Phase 10A. 프롬프트 품질 조율

목표: AI 분석 결과가 발표용 샘플뿐 아니라 실제 OCR 텍스트에서도 안정적으로 나온다.

수정 대상:

- `04_App/server/ai_service.py`
- `04_App/server/models.py`
- `04_App/server/sample.py`
- 검증용 샘플 OCR 텍스트/예상 결과 문서

해야 할 일:

1. 현재 프롬프트를 역할·출력 형식·체크리스트·추출 규칙·다국어 규칙으로 나눠 읽기 쉽게 정리한다.
2. `originalText` 규칙을 더 강하게 만든다. "OCR에 없는 문장을 새로 만들지 말 것", "한국어 원문만 인용할 것", "15~80자 정도의 짧은 근거 문장"을 명시한다.
3. Phase 10을 위해 `sourceQuote` 필드를 추가할지 결정한다. 1차 권장안은 `originalText` 유지 + 서버가 같은 값을 quote로 사용한다. 매칭 품질이 낮으면 그때 `sourceQuote`를 추가한다.
4. 비정상 값·빈 양식·숙식비 공제·교육비 반환·여권 보관·수당 누락 테스트 케이스를 5~7개 만든다.
5. 실제 서버 호출 또는 단위 테스트로 JSON schema, caution 개수, 원문 인용, 번역 key를 확인한다.

완료 기준:

- 빈 양식은 값을 지어내지 않고 `기재 없음`으로 나온다.
- `월 통상임금(100)원`, `100시 35분` 같은 값은 미기재가 아니라 비정상 값으로 나온다.
- `cautionItems[].originalText`가 OCR 원문 일부와 매칭 가능하다.
- `ko`, `en`, 요청 언어의 title/explanation/notice가 모두 채워진다.
- 실패 시 200 샘플이 아니라 502 또는 앱 오류 화면으로 이어진다.

curl http://127.0.0.1:8000/health

## Phase 10B. 동적 다국어 보강

목표: 결과 화면에서 사용자가 어떤 언어를 눌러도 요약값·주의 조항 설명·notice가 영어 fallback으로 남지 않게 한다.

권장 구조:

1. 앱이 결과 화면에서 선택한 언어 key가 있는지 확인한다.
2. key가 없으면 `POST /localize-analysis` 같은 보강 API를 호출한다.
3. 서버는 기존 분석 결과의 한국어 원문·영어 설명·OCR 원문을 바탕으로 target language 필드만 생성한다.
4. 앱은 받은 localized fields를 현재 `AnalysisResult`에 병합한다.
5. 저장 기록에서 열린 결과라면 병합된 내용을 history record에도 저장한다.

수정 대상:

- `04_App/server/main.py`
- `04_App/server/ai_service.py`
- `04_App/server/models.py`
- `04_App/mobile/src/api/contractApi.ts`
- `04_App/mobile/src/app/result.tsx`
- `04_App/mobile/src/data/historyStore.ts`
- `04_App/mobile/src/i18n/localized.ts`

데이터 원칙:

- `originalText`와 `ocrText`는 한국어 OCR 원문 유지.
- 번역 대상은 `summary.*`, `cautionItems[].title`, `cautionItems[].explanation`, `notice`.
- 이미 존재하는 `ko/en` 값은 덮어쓰지 않는다.
- 번역 요청 실패 시 해당 언어 탭에 "번역을 불러오지 못했어요" 상태를 보여주고 영어 fallback임을 숨기지 않는다.

완료 기준:

- 한국어로 분석한 결과에서 네팔어 탭을 누르면 네팔어 동적 내용이 채워진다.
- 영어로 분석한 결과에서 베트남어/크메르어/태국어 탭을 눌러도 설명이 영어로 남지 않는다.
- 저장 기록을 다시 열어도 이미 보강한 언어는 재요청하지 않는다.
- 로딩 중 버튼/탭 상태가 흔들리지 않는다.

## Phase 10C. 결과 UI/UX 정비

목표: 결과 화면이 "AI가 말한 것"이 아니라 "원문을 근거로 사용자가 확인하는 화면"처럼 보이게 한다.

정비 방향:

1. 상단 언어 탭은 너무 많은 언어를 가로 탭으로 모두 보여주는 방식이 부담될 수 있다. 결과 화면에서는 현재 언어 + 변경 버튼 또는 compact picker를 검토한다.
2. 요약 카드, 주의 조항, 원본 사진 영역의 우선순위를 다시 잡는다.
3. `사진에서 확인`이 들어갈 자리를 CautionCard 안에 확보한다.
4. 샘플 배너, 저장됨 안내, 법률 자문 아님 안내가 서로 경쟁하지 않게 정리한다.
5. 결과 화면의 원본 이미지 썸네일은 Phase 10 이후 하이라이트 뷰어 진입과 연결한다.

완료 기준:

- 사용자는 첫 화면에서 핵심 조건과 확인 필요 조항 수를 바로 본다.
- 주의 카드마다 원문 인용과 설명이 분리되어 읽힌다.
- 언어 전환 중 동적 번역 로딩/실패 상태가 명확하다.
- `사진에서 확인`이 결과 카드의 자연스러운 다음 행동으로 보인다.
- 모바일 좁은 화면에서 언어명·버튼·카드 텍스트가 겹치지 않는다.

## Phase 10D. 사진 출처 하이라이트 구현

목표: 주의 조항이 계약서 사진의 어느 영역에서 나온 것인지 직접 확인할 수 있게 한다.

구현 순서:

1. Google Vision OCR 응답에서 page/block/paragraph/word bounding box를 보존한다.
2. OCR 텍스트를 기존처럼 AI에 보내되, 좌표 있는 OCR segment 목록을 서버 내부에 유지한다.
3. AI 결과의 `cautionItems[].originalText`를 OCR segment와 매칭한다.
4. 매칭 성공 시 `source: { pageIndex, quote, boxes, confidence, matchType }`를 붙인다.
5. 앱 `CautionCard`에 `사진에서 확인` 액션을 추가한다.
6. 이미지 뷰어에서 원본 이미지의 contain layout offset을 계산해 하이라이트 박스를 그린다.
7. 낮은 confidence 또는 source 없음이면 버튼을 숨기고 `원문 위치를 정확히 찾기 어려워요`를 표시한다.

완료 기준:

- `월 통상임금(100)원`처럼 짧은 phrase도 원본 사진의 해당 줄 근처를 표시한다.
- 여러 장 업로드 시 올바른 pageIndex로 이동한다.
- 저장 후 다시 열어도 하이라이트가 유지된다.
- 기존 저장 기록처럼 `source`가 없는 결과도 깨지지 않는다.
- 하이라이트가 틀릴 가능성이 있는 항목은 억지로 표시하지 않는다.

## Phase 10E. 데모 QA와 제출 어필 포인트 정리

목표: 구현 결과를 공모전 제출물에서 기술적 차별점으로 설명할 수 있게 한다.

어필 포인트:

- 더미 결과로 숨기지 않고 real OCR/AI 실패를 502와 오류 화면으로 구분했다.
- iPhone 원본/HEIC/대용량 이미지 문제를 업로드 전 JPEG 정규화로 줄였다.
- 백엔드 미도달 문제를 `/health` preflight와 timeout으로 분리했다.
- 언어 변경 후 동적 분석 내용까지 보강하는 구조를 설계했다.
- AI가 좌표를 환각하지 않고 OCR 좌표와 원문 매칭으로 사진 출처를 보여주는 신뢰 UX를 설계했다.

검증 체크리스트:

1. iPhone Safari에서 `http://<server>:8000/health`가 열린다.
2. `EXPO_PUBLIC_API_BASE_URL=http://<server>:8000 npx expo start --go --host lan --clear`로 앱을 실행한다.
3. 실제 계약서 사진 1~2장을 업로드한다.
4. 서버 로그에서 `ocr=real`, `isSample=false`, 업로드 bytes 축소 여부를 확인한다.
5. 결과 화면에서 한국어→영어→대상 언어를 바꾸며 동적 내용이 fallback 없이 표시되는지 확인한다.
6. 주의 카드의 `사진에서 확인`이 올바른 원본 영역으로 이동하는지 확인한다.
7. 실패 케이스에서는 샘플 결과가 아니라 오류 또는 낮은 신뢰도 안내가 뜨는지 확인한다.
