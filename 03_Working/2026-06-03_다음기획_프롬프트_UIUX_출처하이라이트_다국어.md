## 0. 한 줄 결론

다음 개발은 바로 UI를 붙이지 않는다. 먼저 AI 분석 결과가 원문과 맞는지 검증 가능한 구조를 만든다.

이번 세션의 범위는 **Phase 10A: 프롬프트 품질 조율 + `originalText` 검증 기반**이다.

후속 작업인 동적 다국어 보강, 결과 UI 정비, 사진 출처 하이라이트는 다음 세션으로 분리한다.

## 1. 왜 이 순서인가

현재 앱은 OCR → AI 분석 → 결과 표시 → 저장·재열람 → 언어 설정까지 연결됐다. 남은 핵심 문제는 기능 수가 아니라 **신뢰도**다.

사용자가 결과를 믿으려면 다음 질문에 답할 수 있어야 한다.

1. AI가 계약서에 실제로 적힌 내용을 보고 말했나?
2. 주의 조항의 근거 문장이 OCR 원문 안에 있나?
3. 언어를 바꿨을 때 다른 나라 법률 기준을 섞지 않나?
4. 나중에 사진에서 근거 위치를 보여줄 수 있나?

따라서 구현 순서는 다음으로 고정한다.

| 순서 | 작업 | 이유 | 이번 세션 여부 |
|---|---|---|---|
| 1 | 프롬프트 품질 조율 | AI 결과의 기준을 먼저 고정해야 한다 | yes |
| 2 | `originalText` 검증 | 사진 하이라이트의 매칭 기준이다 | yes |
| 3 | golden OCR fixture | 운영 하드코딩 없이 품질을 반복 검증한다 | yes |
| 4 | 동적 다국어 보강 | 결과 언어 변경 시 영어 fallback을 막는다 | no |
| 5 | 결과 UI/UX 정비 | 검증 가능한 결과 화면으로 바꾼다 | no |
| 6 | OCR 좌표 기반 하이라이트 | 원본 사진에서 근거 위치를 보여준다 | no |

## 2. 핵심 제품 판단

### 2.1 AI 법률 판정기가 아니다

이 서비스는 위법 여부, 계약 무효, 법률 판단을 확정하지 않는다. 기본 톤은 다음이다.

- 이해를 돕는다.
- 확인 질문을 제공한다.
- 주의 신호를 알려준다.
- 필요하면 상담을 권장한다.

금지 표현:

- `위법입니다`
- `무효입니다`
- `반드시 받을 수 있습니다`
- `법적으로 문제없습니다`

권장 표현:

- `확인이 필요합니다`
- `서명 전에 다시 확인해 보세요`
- `상담을 권장합니다`
- `계약서 원문과 실제 설명이 같은지 확인하세요`

### 2.2 선택 언어는 설명 언어다

앱의 `language` 선택은 사용자가 이해할 설명 언어를 고르는 기능이다. 계약서에 적용되는 법률 국가를 바꾸는 기능이 아니다.

MVP 기준은 **한국에서 사용하는 한국어 근로계약서**다. 사용자가 베트남어, 네팔어, 태국어 설명을 선택해도 AI는 베트남, 네팔, 태국 노동법을 가져와 판단하면 안 된다.

프롬프트에 반드시 넣을 원칙:

- 선택 언어는 설명 언어일 뿐, 적용 법률 국가를 바꾸지 않는다.
- 분석 기준은 한국에서 사용하는 한국어 근로계약서다.
- 다른 국가의 노동법 위반 여부를 판단하지 않는다.
- 다국어 설명은 한국어 원문과 한국 근로계약서 맥락을 쉽게 이해하도록 돕는 용도다.

제출/발표 스토리라인:

다국어 AI는 언어에 따라 해당 국가의 법률 기준을 섞어 가져올 위험이 있다. 이 프로젝트는 프롬프트와 데이터 구조에서 **언어**와 **법률 기준**을 분리해 그 오류를 줄였다.

### 2.3 사진 출처 하이라이트는 AI 좌표가 아니다

AI가 좌표를 만들면 안 된다. 좌표는 Google Vision OCR 결과의 bounding box에서 나온다.

AI가 해야 할 일:

- `originalText`에 OCR 원문 일부를 짧게 인용한다.
- 설명문과 원문 인용을 섞지 않는다.
- 번역문을 `originalText`에 넣지 않는다.

서버가 해야 할 일:

- OCR segment와 `originalText`를 매칭한다.
- 매칭 성공 시 `source` 좌표를 붙인다.
- 매칭 실패 시 하이라이트를 만들지 않는다.

## 3. 현재 문제

### 3.1 프롬프트 품질 문제

현재 프롬프트는 이미 많은 규칙을 담고 있지만, 역할·체크리스트·추출 규칙·다국어 규칙이 한 덩어리로 섞여 있다.

그 결과 다음 위험이 있다.

- 빈 양식의 라벨을 실제 값처럼 해석할 수 있다.
- `월 통상임금(100)원` 같은 비정상 값을 `기재 없음`으로 뭉갤 수 있다.
- `originalText`가 OCR 원문에 없는 설명문으로 나올 수 있다.
- 선택 언어에 따라 다른 나라 노동법 맥락이 섞일 수 있다.

### 3.2 동적 다국어 fallback 문제

현재 서버 분석은 `ko`, `en`, `요청언어`만 생성한다.

예:

- 한국어로 분석하면 결과 JSON은 보통 `ko`, `en`만 가진다.
- 이후 결과 화면에서 `ne`, `vi`, `km` 같은 언어를 누르면 해당 key가 없다.
- 앱의 `getLocalized()`는 `value[language] ?? value.en ?? value.ko` 순서라서 영어로 fallback한다.

이 문제는 Phase 10B에서 해결한다. 이번 세션에서는 구현하지 않는다.

### 3.3 사진 하이라이트 전제 문제

사진 하이라이트는 UI만 붙이면 실패한다. 먼저 AI가 OCR 원문과 매칭 가능한 `originalText`를 줘야 한다.

이번 세션의 `originalText` 검증이 Phase 10D의 선행 조건이다.

## 4. 이번 세션 범위: Phase 10A

### 목표

실제 OCR 텍스트에서도 AI 분석 결과가 안정적으로 나오고, 각 주의 항목의 `originalText`가 OCR 원문과 매칭 가능하게 만든다.

### 수정 대상

- `04_App/server/ai_service.py`
- 검증용 fixture 또는 스크립트
- 필요 시 `04_App/server/models.py`
- 필요 시 `04_App/server/sample.py`

### 할 일

1. `ai_service.py` 프롬프트를 구조화한다.
   - 역할
   - 출력 JSON 형식
   - 핵심 조건 추출 규칙
   - 주의 조항 체크리스트
   - 빈 양식/비정상 값 판단 규칙
   - `originalText` 인용 규칙
   - 다국어 설명 규칙
   - 설명 언어와 법률 기준 분리 규칙

2. `originalText` 규칙을 강화한다.
   - OCR에 실제로 있는 한국어 원문만 사용한다.
   - AI 설명문, 번역문, 요약문을 넣지 않는다.
   - 가능하면 15~80자 정도로 짧게 둔다.
   - 너무 긴 조항은 핵심 phrase를 인용한다.
   - 원문이 깨졌으면 OCR에 보이는 형태 그대로 인용하고 설명에서 OCR 확인 필요를 말한다.

3. `sourceQuote` 추가 여부를 보류한다.
   - 1차는 `originalText`만 사용한다.
   - 서버가 나중에 `source.quote = originalText`로 사용할 수 있게 한다.
   - 매칭 품질이 낮으면 다음 세션에서 `sourceQuote` 필드를 검토한다.

4. golden OCR fixture를 만든다.
   - 운영 로직에 하드코딩하지 않는다.
   - 테스트/검증용 입력으로만 쓴다.
   - 정답 문장 전체 일치가 아니라, 반드시 잡아야 할 성격을 검증한다.

5. 검증 기준을 만든다.
   - AI JSON schema 통과
   - caution 항목 누락 여부
   - `originalText in ocrText`
   - `ko`, `en`, 요청 언어 key 존재
   - real AI 실패 시 502 흐름 유지

## 5. Golden OCR fixture 설계

fixture는 운영용 샘플 응답이 아니다. 프롬프트 품질을 반복 확인하는 검증 데이터다.

권장 파일:

```text
04_App/server/tests/golden_ocr/
├── blank_contract.txt
├── abnormal_salary.txt
├── abnormal_work_hours.txt
├── housing_deduction.txt
├── training_fee_return.txt
├── passport_retention.txt
└── overtime_allowance_missing.txt
```

검증 관점:

| 케이스 | OCR 예시 | 기대 성격 |
|---|---|---|
| 빈 양식 | `월 통상임금( )원` | 실제 값으로 쓰지 않고 `기재 없음` |
| 비정상 임금 | `월 통상임금(100)원` | 미기재가 아니라 비정상 임금 확인 |
| 비정상 시간 | `100시 35분 ~ 00시 30분` | 미기재가 아니라 비정상 근무시간 확인 |
| 숙식비 공제 | `숙식비 200,000원을 공제한다` | 공제 항목 누락 금지 |
| 교육비 반환 | `1년 이내 퇴사 시 교육비 100만원 반환` | 위약금/반환 조항 확인 |
| 여권 보관 | `여권은 회사가 보관한다` | 사용자에게 불리할 수 있는 조항 확인 |
| 수당 누락 | `연장근로 가능`만 있고 수당 설명 없음 | 연장·야간·휴일수당 확인 |

하지 말아야 할 것:

```python
if "월 통상임금(100)원" in ocr_text:
    return fixed_result
```

해야 할 것:

- OCR 텍스트를 AI에 그대로 넣는다.
- 응답 JSON이 기준을 만족하는지만 확인한다.
- 기준 미달이면 프롬프트를 조율한다.

## 7. 다음 세션 후보

### Phase 10B. 동적 다국어 보강

목표: 결과 화면에서 사용자가 어떤 언어를 눌러도 요약값·주의 조항 설명·notice가 영어 fallback으로 남지 않게 한다.

권장 구조:

1. 앱이 선택 언어 key 존재 여부를 확인한다.
2. key가 없으면 `POST /localize-analysis`를 호출한다.
3. 서버는 기존 분석 결과에서 target language 필드만 생성한다.
4. 앱은 받은 localized fields를 현재 session에 병합한다.
5. 저장 기록에서 열린 결과라면 history record에도 병합한다.

데이터 원칙:

- `originalText`와 `ocrText`는 한국어 OCR 원문 유지
- 번역 대상은 `summary.*`, `cautionItems[].title`, `cautionItems[].explanation`, `notice`
- 이미 존재하는 `ko/en` 값은 덮어쓰지 않음
- 번역 실패 시 영어 fallback임을 숨기지 않음

### Phase 10C. 결과 UI/UX 정비

목표: 결과 화면을 "AI가 말한 것"이 아니라 "원문을 근거로 사용자가 확인하는 화면"처럼 보이게 한다.

현재 결과 화면에서 보이는 문제:

- 샘플 배너, 저장됨 안내, 16개 언어 탭이 먼저 보여서 실제 분석 결과가 아래로 밀린다.
- 16개 언어를 가로 탭으로 모두 노출하면 결과 화면의 주목점이 언어 선택으로 쏠린다.
- `SummaryCard`는 label 폭이 고정이라 긴 번역 라벨에서 깨질 수 있다.
- `CautionCard`는 원문 인용과 설명은 좋지만, 사용자의 다음 행동인 원문 사진 확인으로 이어지는 자리가 없다.
- 원본 사진 썸네일이 화면 하단에 있어 "근거 확인"보다 첨부 이미지처럼 보인다.
- 이미지 모달은 닫기 버튼, 페이지 번호, 근거 설명이 없어 확인 화면이라기보다 단순 이미지 확대에 가깝다.
- 법률 자문 아님 안내, 샘플 안내, 저장 안내가 각각 카드처럼 보여 주의 카드와 시각적으로 경쟁한다.

정비 원칙:

- 첫 화면은 `무엇을 확인해야 하는지`를 먼저 보여준다.
- 언어 선택은 보조 기능으로 낮추고, 번역 로딩/실패 상태는 숨기지 않는다.
- 원문 인용은 AI 설명보다 작은 부속이 아니라 신뢰 UX의 핵심 근거로 다룬다.
- 사진 확인은 원본 이미지 목록에서 출발하지 않고, 각 주의 카드의 근거 행동으로 출발한다.
- 법률 자문 아님 안내는 항상 보이되, 결과 이해를 방해하지 않는 낮은 시각 위계로 둔다.

권장 화면 순서:

```text
Header
→ 결과 요약 hero: 확인 필요 N건 · 분석 기준 한국 근로계약서 · 현재 설명 언어
→ 핵심 조건 요약
→ 확인 필요 조항 목록
   → 각 카드: 레벨 / 제목 / 원문 인용 / 설명 / 사진에서 확인
→ 원본 사진 섹션
→ 참고용 안내 / 저장 상태 / 샘플 상태
```

언어 UI 권장안:

- 결과 화면에서는 16개 탭을 계속 노출하지 않는다.
- 상단 hero에 `설명 언어: 한국어`처럼 현재 언어만 보여준다.
- `변경` 버튼을 누르면 언어 선택 sheet 또는 compact picker를 연다.
- 선택한 언어의 동적 분석 key가 없으면 `번역 불러오는 중` 상태를 보여준다.
- 실패하면 `이 언어 설명을 불러오지 못했어요. 영어 설명을 함께 보여드려요.`처럼 fallback을 드러낸다.

요약 카드 권장안:

- 긴 다국어 라벨을 고려해 2열 고정폭보다 세로형 row를 우선 검토한다.
- `임금`, `근무시간`, `휴일`, `계약기간`, `공제`는 모두 같은 무게로 보이기보다, 비정상/미기재가 있는 항목을 살짝 강조한다.
- 값이 `기재 없음`이면 일반 텍스트와 구분한다. 단, 빨간색으로 단정하지 않고 `확인 필요` 톤으로 둔다.

주의 카드 권장안:

- 카드 안의 우선순위는 `제목 → 쉬운 설명 → 계약서 원문 → 사진에서 확인`보다 `제목 → 계약서 원문 → 쉬운 설명 → 사진에서 확인`이 낫다. 이 서비스의 차별점은 원문 근거이기 때문이다.
- `사진에서 확인`은 source가 high/medium일 때만 표시한다.
- source가 없으면 `사진 위치를 정확히 찾기 어려워요. 원문 인용을 직접 확인해 주세요.`를 작게 표시한다.
- level 배지는 유지하되, `check/review/info`가 법적 판정처럼 보이지 않게 문구는 `꼭 확인`, `다시 보기`, `참고` 계열로 유지한다.

이미지 뷰어 권장안:

- 단순 modal 확대가 아니라 "근거 확인 화면"으로 바꾼다.
- 상단: 닫기, `사진에서 확인`, 페이지 번호
- 중앙: 원본 이미지와 하이라이트
- 하단: 주의 카드 제목, OCR 원문, 짧은 설명
- 하이라이트 박스는 파란색 계열을 쓰고, 위험을 단정하는 빨간 박스는 피한다.

상태 안내 정리:

- 저장됨 안내는 결과 상단의 큰 카드가 아니라 작은 상태 텍스트로 낮춘다.
- 샘플 결과 안내는 필요하지만 실제 결과보다 먼저 크게 보이면 안 된다. 상단 작은 warning strip 또는 하단 참고 영역으로 둔다.
- 법률 자문 아님 안내는 결과 하단에 두되, `분석 기준: 한국 근로계약서 / 설명 언어: 선택 언어 / 법률 자문 아님`을 한 묶음으로 정리한다.

### Phase 10D. 사진 출처 하이라이트

목표: 주의 조항이 계약서 사진의 어느 영역에서 나온 것인지 직접 확인할 수 있게 한다.

구현 순서:

1. Google Vision OCR 응답에서 page/block/paragraph/word bounding box 보존
2. OCR 텍스트는 기존처럼 AI에 전달
3. AI 결과의 `cautionItems[].originalText`를 OCR segment와 매칭
4. 성공 시 `source: { pageIndex, quote, boxes, confidence, matchType }` 추가
5. 앱 이미지 뷰어에서 contain layout 기준으로 하이라이트 표시
6. 낮은 confidence 또는 source 없음이면 버튼 숨김

### Phase 10E. 데모 QA와 제출 스토리

어필 포인트:

- real OCR/AI 실패를 샘플 200으로 숨기지 않고 502와 앱 오류 화면으로 구분했다.
- iPhone 원본/HEIC/대용량 이미지를 업로드 전 JPEG 정규화로 줄였다.
- `/health` preflight와 timeout으로 백엔드 미도달 문제를 분리했다.
- 선택 언어를 설명 언어로 제한하고 법률 기준은 한국 근로계약서로 고정했다.
- 언어 변경 후 동적 분석 내용까지 보강하는 구조를 설계했다.
- AI가 좌표를 만들지 않고 OCR 좌표와 원문 매칭으로 사진 출처를 보여준다.
