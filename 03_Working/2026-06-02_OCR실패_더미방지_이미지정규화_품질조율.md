# 2026-06-02 OCR 실패·더미 결과 방지·이미지 정규화 품질 조율 기록

> 한 줄 결정: **서버가 실제 분석에 실패했을 때 더미 분석 결과를 보여주지 않고 실패를 명확히 드러내며, 촬영 이미지는 OCR에 적합한 JPEG로 정규화해서 실제 분석 성공률을 높인다.**

이 문서는 단순 버그 수정 기록이 아니라, 공모전 보고서/발표에서 설명할 수 있는 **AI 분석 품질 조율 과정**과 **신뢰 UX 개선 포인트**를 남기기 위한 문서다.

---

## 1. 발견된 문제

실기기 테스트 중 서버 로그에는 다음처럼 요청 성공처럼 보이는 기록이 찍혔다.

```text
INFO: 192.168.4.4:54428 - "POST /analyze-contract HTTP/1.1" 200 OK
```

하지만 앱 결과 화면에는 사용자가 촬영한 계약서 분석이 아니라, 데모용 샘플 계약서 결과가 표시됐다.

이 문제는 심각하다. 사용자는 "내 계약서가 분석됐다"고 믿을 수 있는데, 실제로는 샘플 결과를 보고 있을 수 있기 때문이다. 근로계약서처럼 민감하고 중요한 문서를 다루는 서비스에서는, 실패를 보기 좋게 숨기는 것보다 실패를 정확히 알리는 것이 더 안전하다.

이후 샘플 폴백을 막자 실제 실패가 다음처럼 드러났다.

```text
INFO: Application startup complete.
[analyze:error] lang=ko pages=2 bytes=7719227 ocr=real reason=RuntimeError
INFO: 192.168.4.9:63774 - "POST /analyze-contract HTTP/1.1" 502 Bad Gateway
```

즉 문제는 두 단계였다.

1. 예전에는 실제 실패가 `200 OK + 샘플 결과`로 숨겨졌다.
2. 실패를 숨기지 않게 바꾸자, 실제 OCR 실패가 `502 Bad Gateway`로 드러났다.

## 2. 원인 분석

### 원인 A. 서버 `.env` 로딩 위치가 불안정했다

기존 `config.py`는 `load_dotenv()`만 사용했다. 이 방식은 서버를 어느 폴더에서 실행하느냐에 따라 `.env`를 못 읽을 수 있다.

예를 들어 `04_App/server` 밖에서 서버를 실행하면 `OPENAI_API_KEY`, `GOOGLE_VISION_API_KEY`를 못 읽고 샘플 모드로 동작할 수 있었다.

수정:

- `config.py`가 항상 `04_App/server/.env`를 읽도록 변경했다.
- 서버 실행 위치와 관계없이 `/health`에서 OCR/AI 모드가 안정적으로 표시된다.

### 원인 B. 실제 OCR/AI 실패가 샘플 결과로 덮였다

기존 서버는 실제 OCR 호출이 실패하거나 AI 응답이 깨져도 샘플 OCR/샘플 분석 결과로 폴백했다.

이 정책은 발표 데모 안정성에는 좋아 보일 수 있지만, 실제 사용자 테스트에서는 치명적이다. 서버가 실제 분석을 못 했는데도 앱은 정상 결과처럼 보여줄 수 있기 때문이다.

수정:

- 키가 없는 명시적 샘플 모드에서만 샘플 결과를 허용한다.
- 키가 있고 `/health`가 `real`인 상태에서 OCR/AI가 실패하면 샘플로 덮지 않는다.
- 실제 OCR/AI 실패는 `502` 오류로 반환한다.

### 원인 C. 앱도 서버 실패를 로컬 샘플로 덮었다

모바일 앱의 loading 화면도 서버 요청 실패 시 `sampleAnalysis`를 결과로 넣고 결과 화면으로 이동했다.

이 때문에 서버에서 실패를 반환하더라도 앱이 다시 더미 결과로 바꿔버릴 수 있었다.

수정:

- 서버 요청 실패 시 결과 화면으로 이동하지 않는다.
- 앱은 "분석에 실패했어요" 오류 화면을 보여준다.
- 사용자는 다시 촬영하거나 홈으로 돌아갈 수 있다.

### 원인 D. iPhone 원본 이미지가 OCR 입력으로 너무 컸다

실패 로그에서 2장 업로드 크기가 약 7.7MB였다.

```text
pages=2 bytes=7719227
```

iPhone 원본 사진은 해상도와 용량이 크고, 갤러리 선택 시 HEIC 또는 고해상도 JPEG가 그대로 들어올 수 있다. Google Vision OCR은 이미지 자체가 너무 크거나 형식/인코딩이 맞지 않을 때 실패할 수 있다.

수정:

- 촬영/갤러리 이미지를 업로드 전 JPEG로 정규화한다.
- 최대 폭을 `1600px`로 줄인다.
- JPEG 품질을 `0.72`로 압축한다.
- 사용 라이브러리: `expo-image-manipulator`

## 5. 공모전 어필포인트

### 5-1. 실패를 숨기지 않는 AI 서비스

보고서 문장:

> 초기 테스트에서는 서버가 OCR 또는 AI 분석에 실패해도 샘플 결과가 표시되는 문제가 있었다. 이는 데모 안정성에는 유리하지만, 실제 사용자가 자신의 계약서가 분석됐다고 오해할 수 있어 위험하다고 판단했다. 따라서 실제 분석 모드에서는 실패를 샘플로 덮지 않고 오류로 표시하도록 수정해, 사용자가 결과의 신뢰 여부를 명확히 알 수 있게 했다.

### 5-2. 민감 문서 서비스에 맞는 신뢰 UX

보고서 문장:

> 근로계약서는 임금, 근무시간, 공제, 퇴사 조건 등 민감한 정보를 포함하므로, 분석 실패를 그럴듯한 결과로 대체하는 것은 사용자에게 잘못된 판단을 유도할 수 있다. 본 프로젝트는 실패를 명확히 드러내고 재촬영을 안내하는 방식으로, AI 결과를 무조건 믿게 하기보다 사용자가 분석 상태를 확인할 수 있는 신뢰 UX를 설계했다.

### 5-3. 실제 촬영 환경을 반영한 입력 품질 개선

보고서 문장:

> 실기기 테스트에서 iPhone 원본 사진 2장의 업로드 크기가 약 7.7MB까지 커져 OCR 실패가 발생할 수 있음을 확인했다. 이를 해결하기 위해 촬영·갤러리 이미지를 업로드 전에 최대 폭 1600px의 JPEG로 정규화하고 압축했다. 이 조정은 OCR 입력 품질을 안정화하고, 네트워크 전송 부담과 외부 OCR API 실패 가능성을 줄이기 위한 실제 테스트 기반 개선이다.

### 5-4. AI 품질 조율의 구체성

보고서 문장:

> 단순히 OCR과 AI API를 연결하는 데 그치지 않고, 실제 테스트 중 발생한 실패 유형을 기준으로 품질 조율을 진행했다. 빈 양식과 비정상 입력값을 구분하도록 프롬프트를 조정했고, OCR 실패가 샘플 결과로 숨겨지지 않도록 서버와 앱의 폴백 정책을 분리했다. 또한 이미지 정규화를 적용해 AI 분석 이전 단계인 OCR 입력 품질까지 개선했다.

### 5-5. 데모 안정성과 실제 서비스 안전성의 분리

보고서 문장:

> 발표 환경에서는 샘플 모드가 필요하지만, 실제 분석 모드에서 샘플 결과가 자동으로 섞이면 사용자가 오해할 수 있다. 따라서 `/health`에서 OCR/AI가 sample인지 real인지 확인할 수 있게 하고, 키가 없는 명시적 샘플 모드에서만 샘플 결과를 허용했다. 실제 모드의 실패는 오류로 처리해 데모 안정성과 사용자 안전성을 분리했다.

## 6. 발표에서 설명할 흐름

발표용 설명 순서:

1. 처음에는 서버 로그가 `200 OK`라서 정상처럼 보였지만, 앱에는 사용자가 촬영한 계약서가 아니라 샘플 결과가 표시됐다.
2. 원인을 추적해 보니 서버와 앱 양쪽에서 실패를 샘플 결과로 덮는 구조가 있었다.
3. 근로계약서 서비스에서는 실패를 숨기는 것이 더 위험하다고 판단해, 실제 분석 실패는 오류로 드러내도록 수정했다.
4. 이후 실제 OCR 실패가 `502`로 드러났고, iPhone 원본 이미지가 너무 큰 문제를 확인했다.
5. 촬영 이미지를 업로드 전 JPEG로 정규화하고 압축해 OCR 입력 품질을 개선했다.
6. 결과적으로 이 기능은 단순한 버그 수정이 아니라, AI 분석 결과의 신뢰성과 사용자 안전을 높이는 품질 조율 과정이다.

## 8. 남은 주의점

- 이미지 정규화는 OCR 실패 가능성을 줄이지만, 흐릿한 사진·잘린 문서·심한 그림자는 여전히 실패할 수 있다.
- Phase 10의 `사진에서 확인` 하이라이트는 OCR 좌표 품질에 의존하므로, 이미지 정규화 이후에도 원문 위치 매칭 품질을 별도로 검증해야 한다.
- 실제 보고서에는 API 키, 전체 OCR 원문, 사용자의 계약서 이미지를 넣지 않는다.

## 9. 추가 발견: OCR 단계처럼 보이는 무한 로딩의 실제 원인

2026-06-03 실기기 테스트에서 앱은 열렸지만, 분석 요청 후 OCR 단계에서 매우 오래 멈춘 뒤 "분석에 실패했어요" 화면이 나오는 문제가 다시 발견됐다. 중요한 단서는 **백엔드 터미널에 `/analyze-contract` 로그가 전혀 찍히지 않았다**는 점이다.

확인 결과:

```bash
curl http://127.0.0.1:8000/health
# {"ok":true,"mode":{"ocr":"real","ai":"real"}}

curl --max-time 5 http://192.168.4.37:8000/health
# 응답 없음 / timeout
```

즉 서버 프로세스는 로컬 루프백에서는 살아 있었지만, 앱이 접근하는 LAN 주소(`192.168.4.37:8000`)로는 응답하지 않았다. 이 경우 앱은 OCR이 느린 것처럼 보이지만 실제로는 **백엔드에 도착조차 하지 못한 상태**다.

Mac 방화벽도 켜져 있었다.

```text
Firewall is enabled. (State = 1)
```

### 추가 수정

앱 요청 흐름을 다음처럼 바꿨다.

1. `/analyze-contract` 업로드 전에 `/health`를 먼저 호출한다.
2. `/health`가 6초 안에 응답하지 않으면 바로 오류 화면으로 보낸다.
3. 실제 분석 업로드도 65초 제한 시간을 둔다.
4. 더 이상 OCR 단계에서 무한 대기처럼 보이지 않게 한다.

수정 파일:

- `04_App/mobile/src/api/contractApi.ts`

핵심 의미:

- OCR이 느린 문제와 백엔드 연결 실패를 구분할 수 있다.
- 백엔드에 로그가 없으면 OCR 문제가 아니라 네트워크/방화벽/API 주소 문제로 판단한다.
- 사용자는 오래 기다린 뒤 막연히 실패하는 대신, 백엔드 연결 문제를 빠르게 확인할 수 있다.

보고서 문장:

> 실기기 테스트에서 앱이 OCR 단계에서 오래 멈추는 문제가 있었지만, 백엔드 로그를 확인한 결과 실제 OCR 지연이 아니라 앱이 백엔드 LAN 주소에 도달하지 못하는 네트워크 문제임을 구분했다. 이후 분석 요청 전에 `/health` 사전 확인을 추가하고 요청 타임아웃을 설정해, 사용자가 무한 로딩처럼 느끼지 않도록 개선했다. 이는 AI 분석 품질뿐 아니라 실제 데모·사용 환경에서 장애 원인을 구분하는 안정성 개선이다.

다음 테스트 기준:

- iPhone Safari에서 `http://192.168.4.37:8000/health`가 열려야 한다.
- Mac에서 `curl http://192.168.4.37:8000/health`도 응답해야 한다.
- 둘 중 하나라도 안 되면 앱 분석 요청은 백엔드에 도착하지 않는다.
- 이때는 Mac 방화벽 설정, 같은 Wi-Fi 여부, 서버 실행 옵션(`--host 0.0.0.0`)을 먼저 확인한다.

### 추가 수정: IP 변경 때마다 코드 수정하지 않도록 분리

같은 날 테스트 중 사용자가 확인한 주소가 `112.170.4.114`로 바뀌었다. 이 주소는 `192.168.x.x` 계열 LAN 주소가 아니라 공인 IP일 가능성이 높다. Expo 실기기 테스트에서는 주소의 종류보다 **iPhone Safari에서 실제로 `http://<IP>:8000/health`가 열리는지**가 기준이다.

이 문제는 매번 `04_App/mobile/src/constants/config.ts`를 직접 고치면 반복된다. 그래서 앱 설정을 다음처럼 조정했다.

```ts
const runtimeApiBaseUrl = process.env.EXPO_PUBLIC_API_BASE_URL?.trim();
```

실행할 때:

```bash
EXPO_PUBLIC_API_BASE_URL=http://<iPhone에서-health가-열리는-IP>:8000 npx expo start --go --host lan --clear
```

이렇게 하면 IP가 바뀌어도 코드 수정 없이 Expo 번들 실행 명령에서 백엔드 주소를 바꿀 수 있다. `112.170.4.114`는 아래 명령 또는 iPhone Safari에서 성공할 때만 사용한다.

```bash
curl --max-time 5 http://112.170.4.114:8000/health
```

현재 확인된 상태:

```bash
ipconfig getifaddr en0
# 112.170.4.114

curl --max-time 5 http://112.170.4.114:8000/health
# Failed to connect

curl --max-time 5 http://127.0.0.1:8000/health
# Failed to connect

lsof -nP -iTCP:8000 -sTCP:LISTEN
# no listener
```

따라서 이 시점의 실패는 OCR 문제가 아니라 **백엔드가 실행 중이 아닌 상태**다. 먼저 FastAPI를 `--host 0.0.0.0 --port 8000`으로 다시 켠 뒤, `127.0.0.1`과 `112.170.4.114` 양쪽 `/health`를 재확인해야 한다.

보고서 어필 포인트:

> 실기기 데모 중 네트워크 환경이 바뀌면서 백엔드 접속 IP가 달라지는 문제가 발생했다. 이를 단순 코드 수정으로 처리하지 않고, Expo 실행 시점의 환경변수로 API 주소를 주입하도록 개선했다. 덕분에 같은 앱 코드로 Wi-Fi·공유기·팀원 기기 환경이 바뀌어도 데모를 안정적으로 재현할 수 있으며, `/health` 사전 점검과 결합해 장애 원인을 OCR/AI 품질 문제와 네트워크 연결 문제로 분리할 수 있게 됐다.
