# Phase 9 IA / 다국어 설계

> 상태: 설계 확정 후보. 구현 전 사용자 리뷰 필요.
> 관련: roadmap Phase 9, progress M7, PRD F-13·F-14·F-15
> 기준: 카메라 UX(A)는 이미 구현 완료로 보고, Phase 9는 앱 전역 언어 전환을 1순위로 시작한다.

## 1. 결정 요약

Phase 9의 목표는 "마이페이지를 추가한다"가 아니라, 앱이 일회용 분석 화면처럼 보이지 않게 하면서도 외국인 근로자가 첫 화면부터 앱을 읽을 수 있게 만드는 것이다.

추천 IA는 다음과 같다.

- 홈은 기존 Toss 톤과 "계약서 분석 시작" 중심 구조를 유지한다.
- 언어 전환은 홈 상단에 바로 노출한다.
- 홈 하단에는 최근 분석 1개 또는 "최근 분석 없음" 영역을 조용하게 둔다.
- 마이페이지는 계정/프로필이 아니라 "내 설정" 허브로 둔다.
- 내 기록은 홈의 최근 분석/전체 보기와 마이페이지 양쪽에서 진입할 수 있게 한다.
- 하단 탭바는 이번 MVP 범위에서 쓰지 않는다.

## 2. IA

```text
홈
├─ 언어 선택
├─ 계약서 분석 시작
│  └─ 촬영/검토/분석/결과
├─ 최근 분석 1개
│  └─ 내 기록
└─ 더보기/설정
   └─ 마이페이지
      ├─ 언어 설정
      ├─ 내 기록
      └─ 개인정보/로컬 저장 안내
```

### 홈

홈은 앱의 핵심 행동인 계약서 분석 시작을 가장 크게 유지한다. 언어 전환은 상단에 둔다. 사용자가 한국어를 못 읽을 수 있으므로 언어 버튼은 텍스트만 한국어로 쓰지 않고, `English`, `Tiếng Việt`처럼 현재 언어명을 그 언어 자체로 보여준다.

홈 하단의 최근 분석 영역은 "앱이 저장과 재열람을 지원한다"는 영속감을 주는 역할이다. 기록 목록을 전면에 내세우지 않고, 최근 항목 1개와 전체 보기만 둔다.

### 언어 선택

언어 선택은 전역 설정이다. 결과 화면 탭만 바꾸는 기능이 아니다. 선택 즉시 홈, 촬영, 검토, 로딩, 결과, 기록, 삭제 확인, 오류 문구에 적용된다.

1차 언어 선택지는 다음처럼 잡는다.

- 한국어
- English
- 14개 대상 언어 중 사용자가 고른 언어

국적이나 국가를 묻지 않는다. 사용자는 언어 토글/리스트에서 읽고 싶은 언어를 직접 고른다. 선택한 언어는 앱 전체 UI와 분석 결과에 함께 적용한다. Phase 9에서는 한국어, English, 14개 대상 언어를 모두 앱 표시 언어이자 결과 언어로 구현한다.

14개 대상 언어는 기존 대상 국가 목록에서 도출하되, UI에서는 국가명이 아니라 언어명으로 보여준다.

- Nepali
- Tetum
- Russian
- Mongolian
- Burmese
- Bengali
- Vietnamese
- Uzbek
- Indonesian
- Chinese
- Khmer
- Kyrgyz
- Thai
- Lao

### 마이페이지

마이페이지는 계정 기능이 아니다. MVP에는 로그인, 이름, 회원 정보, 서버 저장이 없다. 따라서 화면 이름과 내용은 "프로필"보다 "내 설정"에 가깝게 잡는다.

구성:

- 언어
  - 앱/결과 언어 선택
- 내 기록
  - 저장된 분석 목록으로 이동
- 개인정보/저장 안내
  - 분석 결과와 계약서 이미지는 이 기기에만 저장
  - 서버는 OCR/AI 분석 중계만 수행
  - 법률 자문이 아닌 참고용 안내

### 내 기록

내 기록 화면은 Phase 8 결과를 유지한다. 다만 진입점은 홈 우상단 단일 칩에서 다음 구조로 바꾼다.

- 홈 하단 최근 분석 영역의 "전체 보기"
- 마이페이지의 "내 기록"

기록 화면 자체를 하단 탭으로 독립시키지는 않는다. MVP의 화면 수와 사용 빈도를 고려하면 탭바는 앱을 무겁게 만든다.

## 3. 다국어 구조

코드가 지저분해지지 않게 다국어를 세 레이어로 분리한다.

### 3.1 앱 UI 문구 번역

대상:

- 홈 문구
- 버튼
- 촬영/검토 안내
- 로딩 단계 문구
- 결과 화면 헤더/라벨
- 기록 화면 라벨
- 삭제 확인/오류/빈 상태 문구
- 개인정보/법률 고지

구조:

```text
src/i18n/
├─ languages.ts
├─ translations/
│  ├─ ko.ts
│  ├─ en.ts
│  └─ vi.ts ...
└─ useI18n.ts
```

화면 컴포넌트는 한국어 문자열을 직접 들고 있지 않고, `t.home.startAnalysis`, `t.result.title` 같은 키를 사용한다.

Phase 9에서는 선택한 언어가 앱 껍데기 문구와 결과 설명에 함께 적용된다. 언어 선택 화면에서는 각 언어명을 해당 언어 자체로 표시한다. 버튼, 헤더, 빈 상태, 삭제 확인, 오류 문구처럼 사용자가 행동을 결정하는 문구도 선택 언어로 바뀌어야 한다.

### 3.2 사용자 언어 설정

대상:

- 선택 언어

구조:

```text
src/data/settingsStore.ts
src/i18n/languageMeta.ts
```

저장은 AsyncStorage를 사용한다. 화면마다 `useState('ko')`로 언어를 따로 들고 있지 않게 한다. 앱 루트 또는 공용 훅에서 현재 설정을 읽고, 모든 화면이 같은 값을 사용한다.

설정 예시:

```ts
type AppLanguage = 'ko' | 'en' | 'vi' | string;

type UserSettings = {
  language: AppLanguage;
};
```

`language`는 앱 껍데기 문구, 분석 요청, 결과 표시 기본값에 모두 쓴다. UI 언어와 결과 언어를 따로 두지 않는다. 사용자가 언어를 바꾸면 앱 전체와 결과 화면이 같은 언어 기준으로 바뀐다.

### 3.3 분석 결과 다국어 데이터

앱 UI 번역 파일에는 계약서 분석 결과를 넣지 않는다. 분석 결과는 서버가 생성한 데이터다.

현재 구조는 `explanationKo`, `explanationEn`, `explanationVi`처럼 언어가 필드명에 박혀 있다. 언어가 늘면 컴포넌트마다 분기문이 생기므로 Phase 9에서 바꾼다.

권장 구조:

```ts
type LocalizedText = Partial<Record<AppLanguage, string>>;

type LocalizedSummary = {
  salary: LocalizedText;
  workHours: LocalizedText;
  holiday: LocalizedText;
  contractPeriod: LocalizedText;
  deduction: LocalizedText;
};

type CautionItem = {
  level: CautionLevel;
  originalText: string;
  title: LocalizedText;
  explanation: LocalizedText;
};

type AnalysisResult = {
  ocrText: string;
  summary: LocalizedSummary;
  cautionItems: CautionItem[];
  notice: LocalizedText;
};
```

화면에서는 `getLocalized(value, language, 'ko')` 같은 유틸로 값을 꺼낸다. 컴포넌트가 `if language === 'vi'` 같은 분기를 알지 않게 한다.

## 4. 원문과 번역 원칙

계약서 원문(`originalText`, `ocrText`)은 기본적으로 한국어 그대로 유지한다. 이유는 사용자가 실제 서명하는 문서의 문장을 확인해야 하기 때문이다.

번역 대상:

- 요약 값
- 주의 조항 제목
- 주의 조항 설명
- 참고 안내문
- UI 라벨과 버튼

원문 유지 대상:

- OCR 전체 텍스트
- 주의 조항의 원문 인용문
- 계약서 이미지

원문 영역의 라벨은 번역한다. 예: "계약서 원문" 라벨은 영어/모국어로 바뀌지만, 인용문 내용은 한국어 원문을 보여준다.

## 5. 서버/API 방향

Phase 9에서는 앱이 분석 요청 시 결과 언어를 함께 보낸다.

```text
POST /analyze-contract
files: 계약서 이미지 N장
language: 결과 기본 언어
```

서버는 최소한 다음 언어의 결과를 돌려준다.

- `ko`
- `en`
- 요청한 `language`

요청 언어가 `ko` 또는 `en`이면 중복 생성하지 않는다. 예를 들어 베트남어를 고른 사용자는 `ko/en/vi`, 네팔어를 고른 사용자는 `ko/en/ne`, 몽골어를 고른 사용자는 `ko/en/mn` 결과를 받는다.

AI 프롬프트는 "법률 판정"이 아니라 "근로계약 이해 보조" 톤을 유지한다. 번역 결과도 위법 여부를 단정하지 않고, 확인 질문과 상담 권유 중심으로 쓴다.

## 6. Phase 9 범위

### 포함

- 홈 상단 언어 전환 진입
- 홈 하단 최근 분석 1개/전체 보기
- 마이페이지 신설
- 언어 설정 로컬 저장
- 앱 전역 UI 문구 i18n 구조 분리
- 한국어/영어/14개 대상 언어 앱 UI 문구 구현
- 한국어/영어/14개 대상 언어 결과 언어 구현
- 홈/촬영/검토/로딩/결과/기록/삭제/오류 문구가 선택 언어를 따르도록 변경
- 분석 결과 모델을 언어 확장 가능한 구조로 변경
- 요약 값, 주의 조항 제목/설명, 참고 안내문을 선택 언어로 표시

### 제외

- 로그인/회원가입
- 서버 DB 저장
- 사용자 이름/프로필 사진
- 상담기관 자동 연결
- 원문 하이라이트(Phase 10)
- 카메라 UX 추가 개편
- UI 언어와 결과 언어를 따로 선택하는 고급 설정

## 7. 구현 순서 제안

1. `settingsStore`와 언어 메타데이터를 만든다.
2. 앱 UI 문구를 `i18n`으로 분리하고, 홈/기록/결과부터 적용한다.
3. 홈에 언어 전환과 최근 분석 영역을 붙인다.
4. 마이페이지를 만든다.
5. 결과 화면의 로컬 언어 상태를 전역 설정으로 바꾼다.
6. 분석 결과 타입을 언어 확장 구조로 바꾼다.
7. 서버 응답 스키마와 샘플 데이터를 새 구조에 맞춘다.
8. 백엔드 프롬프트/API를 결과 언어 기반으로 바꾼다.
9. 저장된 기존 기록 호환 처리를 넣는다.
10. 타입체크, 샘플 모드, 실제 서버 분석 흐름을 검증한다.

## 8. 리스크와 안전장치

- 14개 대상 언어의 앱 UI와 결과를 한 번에 구현하면 번역 품질 검수가 약해질 수 있다.
  - UI 문구는 짧고 구조화된 키로 관리한다. 데모 검수는 대표 언어 몇 개를 골라 실제 흐름 전체를 확인하고, 나머지는 키 누락/폴백/렌더링 깨짐을 자동 점검한다.
- 기존 저장 기록은 예전 결과 구조를 갖고 있을 수 있다.
  - 읽기 시 구버전 결과를 새 구조로 변환하는 migration 유틸을 둔다.
- 앱 UI 문구와 분석 결과 번역이 섞이면 유지보수가 어려워진다.
  - UI i18n과 서버 결과 localization을 다른 폴더/타입으로 분리한다.
- 원문까지 번역하면 실제 계약 문장 확인성이 떨어질 수 있다.
  - 원문은 한국어 유지, 설명과 라벨만 번역한다.
- 법률 판단처럼 보일 수 있다.
  - 모든 언어의 고지와 주의 조항 문구에서 "확인", "상담 권장", "참고 안내" 톤을 유지한다.

## 9. 리뷰 질문

구현 전 확인할 질문은 하나다.

- Phase 9의 언어 정책을 "선택한 언어가 앱 전체 UI와 분석 결과에 모두 적용된다"로 고정할 것인가?

추천은 "예"다. 사용자가 언어를 바꿨는데 버튼과 안내가 다른 언어로 남으면, 실제 대상 사용자가 앱을 끝까지 조작하기 어렵다. 구현 범위는 커지지만 UX 일관성과 대상 사용자 적합성은 이쪽이 맞다.
