# Phase 8 Spec: 분석 결과 저장·재열람 (+ 멀티페이지)

> 상태: REVIEW 대기 (v3 — 멀티페이지 + UX 검수 반영)
> 기준일: 2026-06-01
> 관련: roadmap Phase 8 / `progress.json` M6 · PRD F-12·F-13·§9
> 선행 합의:
> - 저장은 **기기 로컬 전용**(계정·서버 DB 없음), 백엔드는 OCR/AI 프록시 역할만. "서버에 계약 데이터 저장 안 함"은 공모전 프라이버시 어필 포인트(PRD §9 callout).
> - 계약서는 **여러 장일 수 있다** → 이번 라운드에 멀티페이지를 실제 동작까지 구현(촬영→서버→저장 전 구간).

## 1. 배경 · 목표

지금은 (1) 분석 결과가 1회성이고(결과 화면 벗어나면 사라짐), (2) 전 파이프라인이 **1장 전용**이다. 이를 함께 해소한다:

- 여러 장 계약서를 촬영/선택해 한 번에 분석한다.
- 분석 결과(요약·주의조항·이미지들·시각)를 기기에 저장한다. (F-12)
- 홈에서 과거 분석을 목록으로 보고 다시 연다. (F-13)

## 2. 범위

### 포함
- **멀티페이지 캡처** — 촬영/갤러리로 여러 장 추가, 썸네일 스트립에서 삭제
- **멀티이미지 업로드** — 앱이 N장을 서버로 전송
- **서버 장별 OCR 합치기** — 페이지별 OCR 후 텍스트를 합쳐 분석(AI는 텍스트만 받으므로 그대로)
- **로컬 저장** — 결과 + 이미지 N장
- **내역 목록** — 썸네일(1페이지) + "N장" + 날짜 + 핵심요약 + 주의 건수
- **상세 재열람** — 기존 결과 화면 재사용 + 계약서 이미지 N장 보기
- **삭제** — 개별(스와이프) + 전체

### 제외 (이번 단계 아님)
- 페이지 **드래그 재정렬** — 캡처 순서 = 페이지 순서로 고정(연기)
- **이미지 위 조항 하이라이트(탭 확인)** — 서버 OCR 좌표 + 조항↔단어 퍼지 매칭 + 오버레이 UI 필요한 별도 단계
- OCR 전체 텍스트 출처 문장 하이라이트(텍스트 점프) — 위의 경량 버전, 이번엔 안 함(주의조항 원문 인용은 이미 표시됨)
- 마이페이지 진입점 — Phase 9
- 계정 / 서버 저장 / 기기 간 동기화 — 의도적으로 안 함
- 핀치 줌, 이미지 다운스케일 — 선택(필요 시)

## 3. 핵심 결정

| 항목 | 결정 | 비고 |
|---|---|---|
| 페이지 수 | 1장 이상 N장 | 캡처 순서 = 페이지 순서 |
| 저장 위치 | 기기 로컬 전용 | 서버 저장 X (§9 유지·어필 포인트) |
| 기록 데이터 | AsyncStorage (단일 키에 JSON 배열) | 기록 수십 개 규모라 충분 |
| 이미지 | expo-file-system 으로 기록별 폴더에 N장 복사 | 캐시 uri는 사라질 수 있어 복사 필요 |
| 이미지 화질 | 원본 그대로 | 용량 문제 생기면 다운스케일(연기) |
| 서버 분석 | 장별 OCR → 합친 텍스트 1회 분석 | OCR/AI 모듈 자체는 변경 없음 |
| 주의조항 출처 | 기존 `originalText` 인용으로 충분 | 이미 CautionCard 에 표시됨 |
| 저장 시점 | 분석 완료 시 **항상** | 샘플 폴백 등 이미지 없으면 `imagePaths: []` |
| 삭제 | 개별(스와이프) + 전체 | 기록 + 이미지 폴더 함께 제거 |
| 진입점 | 홈 우상단 아이콘 | 마이페이지는 Phase 9 |

## 4. 전체 파이프라인

```
촬영/갤러리 (N장 추가·삭제)
 → session.setImages([uri, …])
 → analyzeContract(uris, lang)        // FormData 에 files 다중 파트
 → 서버 POST /analyze-contract (files: List[UploadFile])
       페이지별 run_ocr(bytes) → 텍스트 합치기 → analyze(합친 텍스트, lang)
 → AnalysisResult (ocrText = 합친 텍스트)
 → historyStore.save(result, uris)    // 이미지 N장 기록 폴더로 복사
 → result 화면 (요약·주의조항 + 계약서 이미지 N장 보기)
```

## 6. 데이터 모델

```ts
// src/data/historyStore.ts
import type { AnalysisResult } from '@/data/sampleAnalysis';

export type HistoryRecord = {
  id: string;             // createdAt 기반 문자열
  createdAt: number;      // epoch ms
  result: AnalysisResult; // 기존 타입 그대로 (summary·cautionItems·notice·ocrText)
  imagePaths: string[];   // documentDirectory 내 페이지 이미지들 (없으면 [])
  isSample?: boolean;     // 분석 실패로 샘플 폴백된 기록 → 목록/상세 '샘플' 배지
};
```

- AsyncStorage 키: `@contract-helper/history`, 값 = `HistoryRecord[]` JSON.
- 이미지: 기록별 폴더 `<documentDirectory>history/<id>/page-<n>.jpg` (폴더 단위 삭제가 쉬움).

## 7. 멀티페이지 캡처 (`select.tsx`)

- 상태: `photoUri: string | null` → `pages: string[]`.
- 촬영: `takePictureAsync` 결과를 `pages` 에 **추가**(촬영 후 화면에 머물러 다음 장 추가 가능).
- 갤러리: `launchImageLibraryAsync({ allowsMultipleSelection: true })` 로 여러 장 선택 → 모두 추가.
- UI: 페이지 ≥1 이면 하단에 **썸네일 스트립**(가로) — 각 썸네일에 삭제(×), "장 추가"(촬영/갤러리), "분석하기 (N장)".
- 분석: `session.setImages(pages)` → `/loading`.
- 가드: 0장이면 분석 비활성. 페이지 상한은 소프트 캡(예: 10장) 안내(선택).

## 8. 전송

```ts
// contractApi.ts
analyzeContract(uris: string[], language: Language): Promise<AnalysisResult>
// FormData: language + uris 각각을 'files' 파트로 append
//   uris.forEach((uri,i)=> form.append('files', { uri, name:`page${i+1}.jpg`, type:'image/jpeg' }))
```

## 9. 서버 (`main.py`)

```python
@app.post("/analyze-contract", response_model=AnalysisResult)
async def analyze_contract(files: List[UploadFile] = File(...), language: str = Form("ko")):
    # (기존 import 에 List 추가: from typing import List)
    texts = [run_ocr(await f.read()) for f in files]
    ocr_text = "\n\n".join(t for t in texts if t)   # 페이지 구분으로 합치기
    return analyze(ocr_text, language)
```

- 로깅은 기존처럼 안전하게(페이지 수·총 바이트·모드만, 원문 X).
- 단일 장도 `files` 길이 1로 동일 경로.

## 10. historyStore 인터페이스

```ts
save(result: AnalysisResult, imageUris: string[]): Promise<HistoryRecord>
//  - id/createdAt 생성, imageUris 를 기록 폴더로 복사해 imagePaths 설정, 배열 맨 앞에 추가
list(): Promise<HistoryRecord[]>      // 최신순(createdAt desc)
get(id: string): Promise<HistoryRecord | null>
remove(id: string): Promise<void>     // 레코드 + 이미지 폴더 삭제
clearAll(): Promise<void>             // 전체 + 이미지 루트 정리
```

- `save`는 **best-effort**: 실패해도 결과 화면 표시는 막지 않음(에러 로그만).
- JSON 파싱 실패/파일 없음을 안전 처리(빈 배열 폴백, idempotent 삭제).

## 11. 저장 흐름 (`loading.tsx`)

```
const uris = session.getImages();        // 분석에 쓴 이미지들(없을 수 있음)
result = await analyzeContract(uris, 'ko');  // 실패 시 기존대로 샘플 폴백
session.setResult(result);
void historyStore.save(result, uris);    // fire-and-forget, 실패해도 흐름 유지
router.replace('/result');
```

## 12. 화면 상세

### 홈 (`index.tsx`)
- 우상단 "내 분석 기록" 아이콘 → `/history`. 히어로·CTA 레이아웃 유지.

### 내역 목록 (`history.tsx`)
- 포커스 시 `list()` 로드.
- 행: `썸네일(imagePaths[0], 없으면 플레이스홀더) + "N장" 배지(2장↑) + 날짜 + 핵심요약(계약기간·월급) + 주의 N건`.
- 탭 → `/result?id=<id>`. 빈 상태 문구. 헤더에 "전체 삭제". 행 스와이프 개별 삭제.

### 상세 (`result.tsx`)
- `useLocalSearchParams<{ id?: string }>()` 분기: `id` 있으면 `get(id)`, 없으면 `session`.
- 기존 요약/주의조항/언어 전환/안내 유지.
- **추가** "계약서 이미지" 섹션 — N장이면 가로 페이저(스와이프)·썸네일, 탭하면 전체화면(페이지 넘김). 이미지 0장이면 섹션 숨김. 핀치 줌은 선택.

## 13. 삭제
- 개별: 행 스와이프 → `remove(id)` → 목록 갱신.
- 전체: 헤더 → 확인 → `clearAll()`.
- 두 경우 이미지 폴더까지 제거.

## 15. 검증 · 성공 기준
- `npx tsc --noEmit` + 번들 통과.
- 멀티페이지 흐름: 2장 촬영/선택 → 스트립 표시·1장 삭제 동작 → 분석 → 서버가 장별 OCR 합쳐 분석 → 결과 → **N장 저장** → 목록에 "N장" + 요약 → 탭 → 상세에서 모든 페이지 보기 → 스와이프 삭제 → 사라짐 → 전체삭제 후 빈 상태.
- 1장도 동일 경로로 동작. 샘플 폴백(이미지 0장)도 목록·재열람이 안 깨짐.
- **앱 재시작 후 목록 유지**(영속성) — F-12·F-13 충족.

## 16. 가정 · 구현 시 확인할 점
- expo-file-system SDK 56 정확한 API(신규 `File`/`Directory`/`Paths` vs legacy 함수형)는 구현 시 공식 문서로 확인 — 동작(복사/삭제/폴더 생성)은 동일.
- `files` 다중 파트가 RN FormData + FastAPI `List[UploadFile]` 로 정상 수신되는지 1회 실측.
- AsyncStorage 단일 키 + 전체 배열 재기록은 기록 수십 개 가정. 수백 개로 커지면 재검토(현재 불필요).
- 권장 구현 순서: ① 멀티 파이프라인(select→api→server) → ② historyStore 저장 → ③ 목록·상세·삭제 UI → ④ 흐름 UX(아래 §17).

## 17. UX 결정 (검수 반영)

멀티페이지·저장으로 새로 생긴 UX 구멍을 반영한다. (대상: 한국어 서툰 외국인 근로자 → 숨은 동작·긴 대기·모호함에 약함)

1. **촬영 확인** (select.tsx) — 찍으면 즉시 스트립에 추가, **썸네일 탭하면 크게 보고 삭제**(흐린 장 걸러내기). "탭하여 확인" 힌트 + 페이지 번호 표시. 다시 찍기는 삭제 후 재촬영(끝에 추가).
2. **로딩 대기** (loading.tsx) — 장수 인지 문구("N장을 읽고 있어요, 장수가 많으면 조금 더 걸려요") + 단계 애니메이션이 **끝난 척 멈추지 않게**(마지막을 무한 진행으로). 실시간 장별 진행률은 범위 외(요청 1회 구조).
3. **결과 후 이동** (result.tsx) — 새 분석이면 "완료"→**홈**, 기록에서 열었으면→**목록**. (지금은 뒤로가기가 카메라로 떨어지는 문제 해결)
4. **저장 피드백 + 진입점** — 분석 직후 "기기에 저장됨" 가벼운 인라인 안내(프라이버시 강점 노출), 홈 진입점은 **아이콘+라벨("내 기록")**.
5. **삭제 발견성** — 목록 스와이프 + 상세 화면에 보이는 "삭제" 버튼(숨은 제스처 보완).
6. **이미지 2차 노출** — 결과의 핵심은 분석. 계약서 원본은 분석 아래 **"계약서 원본 N장 보기"** 로 둠.
7. **작은 것** — 날짜 절대표기(2026.06.01) / 샘플 폴백 기록 "샘플" 배지(`isSample`, loading 폴백 시 설정) / 목록에 "이 기기에만 저장돼요" 한 줄.
