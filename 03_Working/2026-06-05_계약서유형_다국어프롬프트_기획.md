# 계약서 유형 분리와 다국어 프롬프트 기획

> 작성일: 2026-06-05  
> 상태: 구현 전 기준 문서

## 1. 한 줄 결론

네팔 PDF는 네팔어만 지원하기 위한 자료가 아니라, **표준근로계약서가 2개 양식으로 갈라진다는 것을 확인하기 위한 대표 샘플**이다.

제품 구조는 다음처럼 고정한다.

```text
계약서 유형: 제조·건설·서비스 / 농축산·어업
설명 언어: 한국어 + 영어 + 14개 대상 언어 = 총 16개
```

계약서 유형과 설명 언어는 서로 다른 축이다. 네팔 PDF를 기준으로 양식 차이를 분석하되, 실제 설명은 기존 16개 언어 체계를 그대로 유지한다.

## 2. 사용자가 준 자료의 의미

이번에 받은 PDF 2개는 모두 네팔어 병기 양식이지만, 목적은 네팔어 특화가 아니다.

- 언어별 PDF를 전부 제공하기 어려우므로 네팔어판을 대표 사례로 제공한 것이다.
- 핵심은 "제조·건설·서비스" 양식과 "농축산·어업" 양식의 구조 차이다.
- 앱은 사용자가 선택한 언어로 설명하되, 한국어 원문과 한국 표준근로계약서 기준을 유지해야 한다.
- AI가 네팔어판을 봤다고 해서 네팔 법률 기준을 섞거나 네팔어만 우선하는 구조가 되면 안 된다.

따라서 이 자료는 다음 용도로 쓴다.

| 용도 | 사용 여부 | 이유 |
|---|---:|---|
| 양식 유형 구분 기준 | 사용 | 제목, 항목, 주석이 실제로 다르다. |
| 프롬프트 컨텍스트 | 사용 | 업종별로 확인해야 할 항목이 다르다. |
| OCR/AI QA 샘플 | 사용 | 이미지 기반 PDF라 실제 OCR 품질 확인에 좋다. |
| 언어별 학습 데이터 | 제한 사용 | 네팔어만으로 16개 언어 품질을 대표할 수 없다. |
| fine-tuning 데이터 | 사용하지 않음 | MVP에서는 fine-tuning보다 프롬프트와 테스트 기준이 안전하다. |

## 3. 현재 지원 언어 기준

현재 앱 언어 구조는 다음 16개다.

- 한국어
- English
- Nepali
- Tetum
- Russian
- Mongolian
- Burmese
- Bengali
- Vietnamese
- Uzbek
- Indonesian
- Chinese
- Khmer
- Kyrgyz
- Thai
- Lao

문서나 발표에서는 "14개 언어"라고만 쓰면 헷갈릴 수 있다. 정확히는 **한국어·영어 + 14개 대상 언어**다.

분석 요청은 기존 원칙을 유지한다.

- 분석 시 서버는 `ko`, `en`, 요청 언어 결과를 생성한다.
- 결과 화면에서 다른 언어를 선택했는데 해당 key가 없으면 `/localize-analysis`로 필요한 언어만 보강한다.
- `originalText`와 `ocrText`는 한국어 원문을 유지한다.
- 선택 언어는 설명 언어이지, 법률 기준 국가가 아니다.

## 4. 두 계약서 유형의 차이

### 4.1 공통 항목

두 양식 모두 다음 항목은 공통이다.

- 사용자 정보: 업체명, 전화번호, 소재지, 성명, 사업자등록번호
- 근로자 정보: 성명, 생년월일, 본국 주소
- 근로계약기간, 수습기간
- 근로장소
- 업무내용
- 근로시간, 휴게시간, 휴일
- 임금, 임금지급일, 지급방법
- 숙식제공
- 정하지 않은 사항은 근로기준법에 따름
- 사용자/근로자 서명

이 공통 항목은 기존 `summary` 구조와 주의 조항 구조로 계속 처리한다.

### 4.2 제조·건설·서비스 양식

특징:

- 제목은 일반 `표준근로계약서`다.
- 근로시간 영역에 시작/종료 시간, 1일 평균 시간외 근로시간, 교대제 선택지가 들어간다.
- 휴일 선택지가 비교적 일반 근로계약서형이다.
  - 일요일
  - 공휴일 유급/무급
  - 매주 토요일
  - 격주 토요일
  - 기타
- 임금 항목은 월 통상임금, 기본급, 고정적 수당, 상여금, 수습기간 중 임금, 연장·야간·휴일근로 수당으로 나뉜다.
- 가사서비스업/개인간병인 관련 예외 안내가 별도 주석으로 보인다.

프롬프트에서 더 세게 봐야 할 항목:

- 교대제와 실제 근로시간이 맞는지
- 연장·야간·휴일근로 수당 설명이 있는지
- 기본급/수당/상여금/수습기간 임금이 서로 모순되지 않는지
- 휴일이 유급인지 무급인지
- 통장·도장 관리 금지 안내와 지급방법
- 숙식비 부담금액

### 4.3 농축산·어업 양식

특징:

- 제목에 `표준근로계약서(농업·축산업·어업 분야)`가 명시된다.
- 업무내용에 재배작물 종류, 가축 종류, 연근해어업, 양식어업, 소금채취업 등 구체적인 일의 종류를 쓰도록 되어 있다.
- 근로시간 영역에 농번기/농한기, 어업의 성어기/휴어기처럼 변동 가능성이 큰 상황이 언급된다.
- 휴일도 일반 휴일뿐 아니라 농번기/성어기 휴일 조정 가능성이 같이 나온다.
- 오른쪽 주석에 농림·축산·양잠·수산 사업의 근로시간·휴게·휴일 규정 적용 예외 취지가 보인다.

프롬프트에서 더 세게 봐야 할 항목:

- 업무내용이 구체적으로 적혔는지
  - 재배작물
  - 가축 종류
  - 어업 종류
  - 양식/소금채취 등
- 농번기/성어기 등으로 근로시간과 휴일이 달라질 수 있다는 점을 사용자가 이해하는지
- 근로시간이 비어 있거나 너무 넓게 적혀 있지 않은지
- 휴일이 정기적으로 부여되는지, 조정 방식이 사용자가 이해할 수 있게 적혀 있는지
- 숙식 제공과 비용 부담이 명확한지
- 임금 항목이 시간급/일급/월급 중 무엇인지

주의: 농축산·어업 양식은 일반 근로시간 기준만으로 단정 판단하면 안 된다. 설명은 "업종 특성상 기준이 다를 수 있으니 실제 근무시간과 휴일을 꼭 확인하세요"로 가야 한다.

## 5. 제품 UX 결정

### 5.1 사용자가 먼저 선택하게 한다

분석 전에 사용자가 둘 중 하나를 선택한다.

```text
계약서 종류

[제조·건설·서비스]  [농축산·어업]

계약서 제목이 다르면 분석 결과에서 다시 알려드릴게요.
```

권장 위치:

- `/select` 촬영/검토 화면
- 분석하기 버튼 바로 위
- 사진을 찍거나 고른 뒤, 최종 분석 전에 선택

이 위치가 좋은 이유:

- 홈을 복잡하게 만들지 않는다.
- 사용자가 계약서 사진을 본 뒤 선택할 수 있다.
- 기존 촬영/갤러리 흐름을 크게 흔들지 않는다.
- API 호출 직전에 세션에 넣기 쉽다.

### 5.2 자동 감지는 보조로만 쓴다

OCR 제목에서 다음 문구를 보면 서버가 유형을 보조 판별할 수 있다.

- `농업`
- `축산업`
- `어업`
- `농업·축산업·어업 분야`
- `농림`
- `수산`

하지만 MVP에서는 자동 감지를 사용자의 선택보다 앞세우지 않는다.

권장 동작:

- 사용자가 `농축산·어업`을 선택했고 OCR 제목도 농축산·어업이면 그대로 분석한다.
- 사용자가 `제조·건설·서비스`를 선택했는데 OCR 제목에 농축산·어업이 보이면 결과 notice나 caution에 "계약서 제목상 농축산·어업 양식으로 보입니다. 선택한 종류와 다를 수 있어요."를 안내한다.
- OCR이 불명확하면 선택값을 그대로 쓰되, 단정하지 않는다.

## 6. API 설계

### 6.1 분석 요청

현재:

```http
POST /analyze-contract
files: page-1.jpg, page-2.jpg, ...
language: ko | en | ne | ...
```

변경:

```http
POST /analyze-contract
files: page-1.jpg, page-2.jpg, ...
language: ko | en | ne | tet | ru | mn | my | bn | vi | uz | id | zh | km | ky | th | lo
contractType: manufacturing_construction_service | agriculture_livestock_fishery
```

`contractType` 이름을 추천한다. `industryType`보다 계약서 양식 자체를 더 정확히 말한다.

### 6.2 서버 모델

1차 구현에서는 `AnalysisResult` 응답 스키마를 바꾸지 않는다.

이유:

- 결과 저장, 로컬 히스토리, 다국어 병합, 샘플 데이터까지 연쇄 수정이 커진다.
- 이번 목적은 먼저 프롬프트 분기 품질을 올리는 것이다.
- 사용자가 나중에 기록에서 "어떤 유형으로 분석했는지"를 봐야 한다면 2차에서 저장 모델에 추가한다.

1차 변경 범위:

- 모바일 `session`에 `contractType` 임시 저장
- `analyzeContract(imageUris, language, contractType)`로 API 호출
- FastAPI `contract_type: str = Form(...)` 수신
- `analyze(ocr_text, language, regions, contract_type)`로 전달
- 프롬프트에 계약서 유형별 기준 블록 추가

2차 후보:

- `AnalysisResult.contractType`
- `AnalysisResult.detectedContractType`
- `AnalysisResult.contractTypeMismatch`
- 기록 목록에서 유형 표시

2차는 데모 안정화 후 결정한다.

## 7. 프롬프트 설계

프롬프트는 다음 3개 층으로 나눈다.

```text
공통 안전 원칙
→ 설명 언어 원칙
→ 계약서 유형별 체크리스트
```

### 7.1 공통 안전 원칙

항상 적용한다.

- 법률 판단, 위법 여부, 계약 무효 여부를 확정하지 않는다.
- 원문에 실제로 적힌 값과 빈 양식의 라벨을 구분한다.
- 사용자가 스스로 확인할 수 있게 설명한다.
- `originalText`는 한국어 OCR 원문 일부만 넣는다.
- 사진 하이라이트를 위해 `sourceRegionIds`를 가능한 한 채운다.

### 7.2 설명 언어 원칙

항상 적용한다.

- `language`는 설명 언어다.
- 분석 기준은 한국에서 사용하는 한국어 근로계약서다.
- 네팔어, 베트남어, 태국어 등으로 설명하더라도 해당 국가의 노동법 기준을 섞지 않는다.
- `ko`, `en`, 요청 언어를 채운다.
- `originalText`는 번역하지 않는다.

### 7.3 계약서 유형별 체크리스트

`contractType=manufacturing_construction_service`일 때:

- 월 통상임금, 기본급, 고정 수당, 상여금, 수습기간 임금
- 연장·야간·휴일근로 수당
- 교대제
- 1일 평균 시간외 근로시간
- 휴일 유급/무급
- 가사서비스업/개인간병인 예외 문구가 보이면 단정하지 말고 별도 확인 안내

`contractType=agriculture_livestock_fishery`일 때:

- 재배작물, 가축 종류, 어업 종류, 양식업, 소금채취업 등 구체 업무
- 농번기/농한기, 성어기/휴어기
- 근로시간·휴게·휴일 규정 예외 취지
- 정기 휴일과 바쁜 시기 휴일 조정
- 숙식 제공과 비용 부담
- 시간급/일급/월급 중 어떤 임금 방식인지

## 8. 놓치기 쉬운 위험

### 8.1 네팔 PDF만 보고 네팔어 특화로 오해할 위험

대응:

- 문서와 코드 주석에 "네팔 PDF는 대표 양식 샘플"이라고 명시한다.
- 언어 지원은 기존 16개 그대로 유지한다.

### 8.2 계약서 유형과 설명 언어가 섞일 위험

나쁜 구조:

```text
네팔어 PDF → 네팔 근로자용 분석 → 네팔 법률/네팔 기준 섞임
```

좋은 구조:

```text
한국 표준근로계약서 유형 → 한국 계약서 항목 분석 → 사용자가 고른 언어로 설명
```

### 8.3 농축산·어업을 일반 근로시간 기준으로 과하게 경고할 위험

농축산·어업 양식은 근로시간·휴게·휴일 관련 주석이 다르다. 이 부분은 단정 경고가 아니라 "업종 특성상 기준이 다를 수 있으니 실제 근무표와 휴일을 확인하세요"로 설명한다.

### 8.4 사용자가 잘못 선택할 위험

대응:

- OCR 제목 기반 보조 감지를 둔다.
- 선택값과 OCR 제목이 다르면 결과에서 조용히 알려준다.
- 분석 자체를 막지는 않는다.

### 8.5 이미지 기반 PDF라 OCR 품질이 불안정할 위험

이번 PDF는 `pdftotext`로 텍스트가 추출되지 않는 이미지 기반 PDF다. 실제 앱과 같은 OCR 경로로 검증해야 한다.

대응:

- PDF를 이미지로 렌더링해 Google Vision OCR fixture로 사용한다.
- 한국어 항목명 기준으로 유형 감지를 우선한다.
- 네팔어 OCR 품질에 의존하지 않는다.

### 8.6 언어 16개 전체를 매번 생성해 비용이 커질 위험

대응:

- 분석 시 `ko`, `en`, 요청 언어만 생성한다.
- 결과 화면에서 다른 언어가 필요할 때만 `/localize-analysis`로 보강한다.
- 이 정책은 기존 Phase 9/12 결정과 같다.

## 9. 구현 순서

### Task 1. 계약서 유형 상수 추가

모바일과 서버에서 같은 두 값을 사용한다.

```ts
type ContractType =
  | 'manufacturing_construction_service'
  | 'agriculture_livestock_fishery';
```

### Task 2. 촬영/검토 화면에 선택 UI 추가

- 선택값이 분석 요청까지 유지된다.

### Task 3. 세션과 API payload 연결

### Task 4. FastAPI 수신

수정 후보:

- `04_App/server/ai_service.py`

- 로그에는 계약서 원문을 남기지 않고 유형 값만 남긴다.

### Task 5. 프롬프트 분기

수정 후보:

- `04_App/server/ai_service.py`

완료 기준:

- 공통 규칙은 유지한다.
- 계약서 유형별 체크리스트가 프롬프트에 추가된다.
- 농축산·어업은 근로시간/휴일을 일반 양식처럼 단정하지 않는다.
- 제조·건설·서비스는 교대제/연장·야간·휴일수당/임금 세부 항목을 더 강하게 본다.

### Task 6. OCR 제목 기반 보조 감지

1차에서는 프롬프트 내부 규칙으로 시작하고, 필요 시 서버 함수로 분리한다.

완료 기준:

- OCR 텍스트에 농축산·어업 제목이 보이면 AI가 mismatch 안내를 만들 수 있다.
- 선택값과 OCR 제목이 다를 때 법률 판정이 아니라 "선택한 종류와 다를 수 있음"으로 안내한다.

### Task 7. 테스트 추가

수정 후보:

- `04_App/server/tests/test_ai_service_validation.py`
- `04_App/server/tests/golden_ocr/`

완료 기준:

- 프롬프트에 계약서 유형별 기준 문구가 들어가는지 확인한다.
- `contractType=agriculture_livestock_fishery`에서 농번기/성어기/업무 세부기재 안내가 들어간다.
- `contractType=manufacturing_construction_service`에서 교대제/연장·야간·휴일수당 안내가 들어간다.
- 기존 16개 언어 validation은 깨지지 않는다.

## 10. 발표/신청서 어필 포인트

이 결정은 단순 UI 옵션이 아니라 공모전 설득 포인트다.

좋은 표현:

> 표준근로계약서는 업종에 따라 양식과 확인해야 할 항목이 다릅니다. 본 서비스는 사용자가 계약서 종류를 선택하면, OCR과 AI가 해당 양식의 구조를 반영해 임금, 근로시간, 휴일, 숙식비, 주의 조항을 설명합니다. 설명은 한국어·영어와 주요 14개 대상 언어로 제공하되, 법률 기준은 한국 표준근로계약서 맥락으로 고정해 잘못된 국가별 법률 혼합을 줄입니다.

피해야 할 표현:

- 업종별 법률 위반을 자동 판정한다.
- 네팔어 계약서를 학습해 모든 외국인 계약서를 판단한다.
- 16개 언어별 법률 기준으로 분석한다.
- 농축산·어업의 장시간 근로를 불법으로 판정한다.

## 11. 다음 결정

구현 전에 확정할 것:

1. UI 문구 최종안
   - `계약서 종류`
   - `제조·건설·서비스`
   - `농축산·어업`
   - `계약서 제목이 다르면 분석 결과에서 다시 알려드릴게요.`
2. `contractType`을 결과 기록에 저장할지 여부
   - 1차는 저장하지 않는 쪽 권장
   - 기록 목록에서 유형 표시가 필요해지면 2차 반영
3. OCR 제목 mismatch를 caution item으로 낼지 notice에 넣을지
   - 1차는 notice 또는 info caution 권장
