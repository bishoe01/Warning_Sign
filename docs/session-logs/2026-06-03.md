## 한 일

프롬프트 품질 조율과 동적 다국어 fallback 해결을 구현했다. 서버는 `/localize-analysis`에서 기존 분석 결과와 `targetLanguage`를 받아 target 언어 field만 생성하고, 앱은 결과 화면에서 언어 key 누락을 감지해 session/history record에 병합 캐시한다.

분석 로딩 UX도 수정했다. 기존에는 0.9초마다 4칸이 모두 차서 대기 시간이 길게 느껴졌고, 이제 완료 전에는 3칸까지만 천천히 진행하다가 실제 분석 완료 후 마지막 칸을 채우고 결과 화면으로 이동한다.

검증은 서버 unittest/py_compile, 모바일 TypeScript, localization merge 스모크 테스트, loading progress 스모크 테스트, `progress.json` JSON 검증까지 통과했다. FastAPI TestClient로 `/localize-analysis` 실제 호출도 200 응답을 확인했다.

## 책 메모

로딩 진행 바는 “처리 단계 설명”보다 “기다림을 어떻게 느끼게 하는가”가 중요하다. 완료 전에 마지막 칸을 채우면 사용자는 완료됐다고 받아들이므로, 마지막 칸은 실제 완료 신호로 남겨두는 편이 낫다.

동적 다국어는 분석 단계의 범위를 늘리는 문제가 아니라 저장된 결과의 cache invalidation 문제에 가깝다. 분석 결과 전체를 다시 만들지 말고, 부족한 localized field만 patch로 보강하는 방식이 유지보수와 비용 면에서 안전하다.
